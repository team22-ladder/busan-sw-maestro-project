## 1. 신분변경(status_change) 해결 전략 ⭐

### 1-1. 실패의 두 가지 근본 원인 (v70 G2_sc 8건 실측 분리)

v70에서 status_change 계열 8건이 모두 실패했는데, 원인은 **하나가 아니라 둘**이다:

| # | 질문 | 실측 | 근본 원인 |
|---|------|:----:|-----------|
| sc01 | 미국 관광비자로 입국 후 **취업으로 바꿀** 수 있나요? | RAG | **A. 키워드 미등록** — "바꾸다"가 EXCEPTION_KEYWORDS에 없음 → US+취업 추출되어 RAG 직행 |
| sc05 | 미국에서 비자 신분을 H-1B로 **조정**하고 싶어요 | RAG | **A. 키워드 미등록** — "조정(adjustment)" 미등록 |
| sc02 | **학생비자에서 취업비자로 전환**하고 싶어요 | RESP | **B. 국가 부재** — 국가 없음 → 재질문, EXC 도달 못 함 |
| sc03 | 관광비자를 학생비자로 **변경** 가능한가요? | RESP | **B. 국가 부재** + "변경" 단독 미매칭 |
| sc04 | 워킹홀리데이에서 정식 취업비자로 **전환**하려면? | RESP | **B. 국가 부재** |
| ext02 | 일본에서 3개월 더 **머물고 싶어요** | RAG | **C. 약한 시간 신호** — "머물다"가 연장 신호로 약함 |

→ **원인 A (키워드/패턴 미등록)**, **원인 B (국가 없으면 EXC 못 감)**, **원인 C (약한 시간 신호)** 세 갈래.

### 1-2. 전략 A — "단어 매칭"에서 "전환 패턴 인식"으로

현재는 `EXCEPTION_KEYWORDS` 부분문자열 매칭에 의존 → "비자 전환"은 잡지만
"취업비자**로 전환**", "학생비자**에서** 취업비자**로**"는 놓침. 또한 코드 주석에도
*"광범위한 키워드(전환 가능 등)는 false-positive 위험"* 이라 명시돼 있어, 단순 키워드 추가는
부작용(유학→취업 *계획* 질문까지 예외로 오분류)을 키운다.

**제안: 구조적 전이 패턴(regex)으로 의도를 식별**한다. 이미 화살표 패턴(`A → B`)은 있으니,
이를 자연어 전이로 일반화한다:

```
[A]비자(에서|를|로|에)\s*…\s*[B]비자?(로|으로)\s*(전환|변경|바꾸|조정|돌리|스위치)
[A](관광|학생|유학|워홀|취업)\s*(후|뒤|다음에|이후)\s*(취업|일|학업|이민)   # "관광 후 취업"
비자\s*(신분|유형|종류|상태)\s*(변경|전환|조정)
```

핵심: **전이 동사(전환/변경/바꾸/조정)가 두 비자/목적 토큰 사이에 있을 때만** status_change로
판정 → "유학하고 싶어요"(신규) 와 "유학에서 취업으로 바꾸고 싶어요"(전환)를 구조로 구분.

**전이 동사 사전(확장 후보)**: 바꾸다/바꿔/변경/전환/조정(adjustment)/돌리다/갈아타다/
스위치/업그레이드 + 영문 change/switch/convert/adjust status.

### 1-3. 전략 B — status_change는 "국가 없어도" 예외로 라우팅

현재 라우팅: `country & purpose → RAG`, 아니면 `RESP(재질문)`. 그래서 국가 없는
신분변경("학생→취업 전환")은 RESP로 빠진다. 그러나 **신분변경은 국가가 없어도
'일반 절차 안내 + 국가 되묻기'가 가능한 예외 도메인**이다.

**제안 라우팅 우선순위 변경(설계)**:
```
is_visa_related == False        → general_chat
status_change 패턴 감지         → exception_handler   ← 국가 유무와 무관하게 우선
country & purpose               → visa_rag_search
(그 외 예외)                    → exception_handler
부족                            → response_formatter(재질문)
```
즉 **status_change 감지를 country 체크보다 앞에** 두고, `exception_handler`가
국가가 없으면 "어느 나라 기준인지" 한 줄 되물으면서도 **전환 일반 절차(신분조정 vs 재발급,
출국 후 재신청 여부 등)**를 먼저 안내하게 한다.

### 1-4. 전략 C — 약한 시간/연장 신호 보강

"3개월 더 머물고 싶어요", "비자 끝나가는데", "곧 만료되는데" 같은 표현은 extension 신호다.
**시간 잔여 + 체류 지속 욕구 패턴**을 추가:
```
(N개월|N일|조금|좀)\s*더\s*(머물|체류|있고)
(비자|체류).{0,6}(끝나|만료|곧 끝|마감)
오버스테이 직전 / 기한 넘기
```

### 1-5. 전략 D — 측정 설계 자체의 교정

v70의 0%는 **테스트가 국가를 일부러 뺀** 탓도 크다. 전략 B를 적용하면 국가 없는 케이스도
EXC로 잡히지만, **검증 데이터는 "국가 명시 / 국가 생략" 두 변형을 모두** 포함해야
실사용 신뢰도와 설계 한계를 분리 측정할 수 있다 (→ §4 데이터셋에 반영).

### 1-6. 부작용 가드 (오버피팅 방지)
- "유학 후 취업하고 싶어요"(전이 동사 없음) → **status_change 아님**(장기 계획). 전이 동사가
  있을 때만 전환으로 판정 → 신규 추천(study/employment) 정상 유지 확인 필요.
- 전략 적용 후 **기본 ABC 100% 회귀 없는지** 반드시 회귀 검증(§4 Group G).

## 2. 그 밖의 저신뢰 영역 탐색

### 2-1. 체류연장 / 장기체류 (각 80%, v180)
- **증상**: "더 있고 싶다", "눌러앉고 싶다", "장기로 살고 싶다" 등 **구어·완곡 표현**에서
  목적(long_stay)·예외(extension) 신호가 약함.
- **경계 혼동**: long_stay(장기 거주 비자 추천) vs extension(현 체류 연장)이 자연어에서 섞임.
  "오래 살고 싶어요"=long_stay, "지금 비자로 더 있고 싶어요"=extension — **현 체류 보유
  여부**가 분기 키.

### 2-2. off-topic 경계 (87%, v180)
- **증상**: 비자 *주변* 질문에서 general_chat vs visa 경계 모호.
  - 비자 무관처럼 보이나 비자 관련: "해외에서 운전하려면?", "국제 운전면허", "해외 계좌 개설"
  - 비자처럼 보이나 무관: "비행기표 싸게 사는 법", "환전 어디서", "해외 유심"
- **전략 방향**: 도메인 키워드(VISA_KEYWORDS) 단순 매칭의 양방향 오류 → "이동·체류 권한"
  관련성 기준으로 경계 재정의(데이터로 측정 먼저).

### 2-3. 후속질문 압축 — 신규·경량 검증 영역 (R11.1, 6/6)
6턴만 검증돼 **다음 리스크가 미검증**:
1. **깊은 체인 안정성**: 10턴+ 후속에서 `is_followup`이 흔들리지 않고 국가 앵커를 유지하는가.
2. **누락 정보 후속**: 후속 질문이 **직전 답변에 없던 정보**를 물을 때(예: 직전엔 서류만
   안내했는데 "비용은?") — 압축하면서도 **필요 정보를 빠뜨리지 않는가**(RAG로 보강하는가).
3. **신규 주제 가드**: 같은 세션 내 국가/목적 전환 시 압축하지 않고 전체 추천으로 복귀하는가
   (v70 T5에서 1건 확인 → 더 다양한 전환 케이스 필요).
4. **세션 간 격리**: 새 세션에서 첫 질문이 후속처럼 생겨도(예: "필요 서류는?") **상위 대화가
   없으므로** 후속 압축이 아니라 재질문/일반 안내로 가야 한다(교차 오염 방지).
5. **거짓 후속 회피**: 같은 국가지만 **다른 비자**로 바뀌는 질문이 후속으로 오분류되지 않는가.

## 3. 다중 대화 세션 검증 전략 (압축 응답 포함)

| 검증축 | 가설 | 측정 방법 |
|--------|------|-----------|
| **압축 정확성** | 후속 → 템플릿 미반복 + 상위 참조 | `## 추천비자` 미출력 + "앞서/안내드린" 포함 + 길이 감소율 |
| **신규 주제 가드** | 국가/목적 전환 → full 복귀 | is_followup=False + 템플릿 재출력 |
| **세션 간 격리** | 새 세션 첫 질문은 후속 아님 | 별도 sid·빈 history → is_followup=False |
| **누락 정보 보강** | 직전에 없던 정보 후속 → 압축이되 내용 충실 | 핵심 키워드(비용/기간/숫자) 실제 포함 여부 |
| **깊은 체인 안정** | 10턴+ 후속 동안 국가 앵커 유지 | 매 턴 국가 일치 + is_followup 유지 |
| **거짓 후속 회피** | 같은 국가·다른 비자 → 신규 취급 | 비자 종류 전환 시 full 복귀 |

검증은 **세션별 독립 sid**로 수행하고, 세션 간 history를 섞지 않는다(격리 자체가 검증 대상).

## 4. 최종 검증 데이터셋 명세 (저신뢰 영역 중심)

### Group M — 다중세션·압축 (24)
- **M1 압축 정확성** 체인(6): 독일/캐나다 비자 후속 → 압축 기대

- **M3 세션 간 격리**(4): 새 sid 첫 질문이 후속형 → 후속 아님 기대

- **M5 깊은 체인**(1×10턴): 국가 앵커 유지 + is_followup 안정
- **M6 거짓 후속 회피**(2): 같은 국가 다른 비자 → 신규 기대

### Group G — 회귀 가드 (6)
- 기본 ABC(취업/유학/여행) → 전략 적용 후에도 100% 유지 확인

## 5. 실행 순서 제안 (다음 단계)
1. 본 데이터셋으로 **현재(미수정) 베이스라인** 측정 → 실패 분포 확정.
2. 전략 A·B·C 구현(별도 작업) → 동일 데이터셋 재측정.
3. **전·후 비교를 `/analytics`에 R12로 추가** → 신분변경 개선 폭 시각화.
4. 회귀 가드(Group G) 통과 + 압축군(Group M) 무회귀 확인 후 확정.
