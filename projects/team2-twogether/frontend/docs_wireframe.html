SECTION 00

0개요 & 설계 전제

이 문서는 기획서의 핵심 기능과 워크플로우를 프론트엔드가 구현 가능한 화면 단위로 분해한 것입니다. 서비스 자체는 단발성 추천(세션 단위)이라 화면 수는 적지만, 에이전트가 도중에 끼어드는 두 분기(확인 질문 / 재검색)를 어떻게 화면으로 표현하느냐가 설계의 중심입니다.

설계 전제 1 · 프론트 형태 기획서는 프론트로 Streamlit을 명시했지만, "프론트엔드 개발자 2명"이라는 전제에 맞춰 본 문서는 FastAPI /recommend 엔드포인트를 호출하는 웹 SPA(React/Vue 등) 기준으로 작성했습니다. 단, 모든 화면은 단일 페이지 안의 상태 전환으로 구성되어 Streamlit으로도 1:1 매핑이 가능합니다. (라우팅이 아닌 viewState 전환 방식)
설계 전제 2 · 범위 한정 MVP 제외 항목(로그인/인증, 장기 메모리, 추천 이력, 멘토 실제 연락·매칭, 멀티유저)은 화면에서도 제외합니다. 따라서 마이페이지·로그인·히스토리 화면은 없습니다. 프론트는 입력 → 진행 → (분기) → 결과 한 흐름에만 집중합니다.

SECTION 02

2사용자 화면 플로우

백엔드 LangGraph 노드 흐름을 사용자가 실제로 보는 화면 전환으로 압축한 다이어그램입니다. 분기는 모두 백엔드 응답의 status 값으로 결정되며, 프론트는 그 값에 따라 화면을 전환합니다.

제출 후 결과 수신 전까지의 대기 화면. 단순 스피너 대신 에이전트 파이프라인 단계를 보여줘 "AI가 생각하는 과정"을 체감시키는 것이 데모 임팩트의 핵심.

No요소설명

구현 방식 (택1)

  • A. 가짜 단계 진행 — 백엔드가 단일 응답만 줄 경우, 프론트가 정해진 타이밍으로 단계 라벨을 순차 점등(가장 단순, MVP 권장).
  • B. 실제 스트리밍 — 백엔드가 SSE/단계 이벤트를 보내면 그대로 반영(여유 시 고도화, P1).

SECTION 05

5API 연동 규약 (프론트 관점)

프론트는 단일 엔드포인트 POST /recommend만 호출합니다. 응답의 status 하나로 화면 분기가 모두 결정됩니다. 아래 스키마는 프론트-백엔드 합의용 제안이며 실제 필드명은 백엔드와 맞추세요.

요청 (Request)

POST /recommend
{
  "session_id": "uuid-...",  // 신규 또는 확인 질문 후 동일 세션
  "project_text": "웹 백엔드는 어느 정도 했는데 모델 서빙/배포가 막막합니다...",
  "tech_stack": ["FastAPI", "React"],  // 선택
  "stage": "배포 준비",  // 선택
  "clarify_answer": null  // 확인 질문 응답 시에만 값
}

응답 (Response) — 3가지 status

// ① 입력 부족 → S-03 확인 질문
{ "status": "need_clarification",
  "question": "가장 걱정되는 부분은 성능·확장성·배포·보안 중 어디에 가깝나요?",
  "options": ["성능","확장성","배포","보안"] }

// ② 정상 추천 → S-04 결과
{ "status": "recommended",
  "gaps": ["모델 서빙", "MLOps", "배포"],
  "refined": false,  // 재검색이 일어났는지
  "mentors": [
    { "name":"가명", "domain":["MLOps"], "keywords":[...],
      "score":92, "reason":"모델 서빙 약점과 직접 연결...",
      "can_help":[...], "less_relevant_for":[...], "profile_summary":"..." }
  ] }

// ③ 재검색 후에도 근거 약함 → S-04 + 제한 배너
{ "status": "limited", "gaps":[...], "mentors":[...],
  "notice": "정확히 맞는 멘토가 부족해 근거가 제한적입니다." }
프론트 분기 규칙 need_clarification → S-03 · recommended/limited → S-04 · mentors가 빈 배열 → S-06 EMPTY · HTTP 오류 → S-06 ERROR. 재검색(low_confidence)은 백엔드 내부에서 자동 처리되므로 프론트는 별도 호출 없이 refined 플래그만 읽으면 됩니다.
백엔드 합의 필요 항목 ① 단계 진행을 스트리밍할지(S-02 방식 A/B 결정) ② session_id 발급 주체(프론트 생성 vs 첫 응답에서 수신) ③ limited를 별도 status로 줄지 recommended + notice로 줄지. — 1주차 초반에 못 박을 것.

SECTION 06

6프론트엔드 2인 작업 분담

의존성을 최소화하려면 "입력·진행 흐름"과 "결과·멘토 표현"으로 가르는 게 가장 깔끔합니다. 둘 사이 계약은 useRecommend() 훅과 응답 스키마뿐이라, 목(mock) 응답으로 병렬 개발이 가능합니다.

A

FE-A · 입력 & 흐름

Input / Loading / Clarify
  • S-01 메인 입력 화면 (textarea + 선택 입력 + 검증)
  • S-02 분석 진행 화면 (StepProgress, 재검색 표시)
  • S-03 확인 질문 화면 (ClarifyQuestion, 답변 병합 재요청)
  • 전역 viewState 상태 머신 / 라우팅
  • 예시 프롬프트 칩, 입력 UX
B

FE-B · 결과 & 멘토

Result / Mentor / States
  • S-04 추천 결과 화면 (약점 요약 + 카드 리스트)
  • S-05 MentorCard · MentorDetailModal
  • S-06 StateView (빈/에러/제한적/타임아웃)
  • 제한적 추천 배너 · 적합도 시각화
  • 모달 접근성(포커스 트랩·ESC)

공동 / 선합의 (둘이 함께 30분이면 충분)

  • API 계약 확정 — 위 4장 스키마를 백엔드와 못 박고, 그 즉시 mock /recommend 응답(3종 status) JSON을 공유 → 백엔드 완성 전 병렬 개발.
  • 공통 컴포넌트/토큰 — <Button> <Chip> + 색·폰트 토큰을 한 파일에 정의(누가 만들지 1명 지정).
  • useRecommend() 훅 — 요청/상태/응답을 캡슐화. 두 사람 모두 이 훅만 통해 통신(직접 fetch 금지).
에이전트 활용 팁 (각자 1 에이전트) ① mock 우선: 에이전트에게 "위 응답 스키마로 3종 status mock JSON과 useRecommend 훅부터 만들어" → 화면을 실데이터처럼 개발. ② 컴포넌트 단위로 위임: 화면 통째보다 MentorCard·StepProgress 같은 단위로 프롬프트하면 결과가 안정적. ③ 이 문서를 컨텍스트로: 해당 화면의 명세 표·와이어프레임 캡처를 에이전트에 붙여 넣으면 요구사항 누락이 줄어듭니다. ④ 스타일은 마지막에 토큰으로 일괄 통일.
