## 1. Executive Summary

본 제품은 플레이어가 제한된 대화 횟수 안에서 여러 용의자와 직접 대화하고, 진술과 증거를 교차검증해 모순을 찾은 뒤 최종 범인을 지목하는 추리 게임이다. 핵심 재미는 단순 선택지가 아니라, 플레이어가 자연어로 캐릭터를 심문하고 용의자의 알리바이, 시간대, 관계, 증거 기록을 연결해 "말의 충돌"을 발견하는 과정에 있다.

공유 UI 시안은 용의자 선택, 대화, 대화 기록, 증거/기록 패널, 추리 노트, 남은 대화 횟수, 최종 범인 지목까지 한 화면에서 순환되는 구조를 보여준다. 이 PRD는 해당 UI를 제품 방향의 기준으로 삼고, PDF 보고서의 이벤트 단위 판단, 신뢰도 기반 평가, 후처리 안정화, 경량 실행 구조를 게임의 AI 판정 엔진과 시스템 설계 원칙으로 반영한다.

MVP 목표는 "한 개의 완성된 사건을 끝까지 플레이할 수 있는 웹 기반 추리 경험"이다. 플레이어는 용의자 4명 이상, 증거 8개 이상, 제한 질문 12회 내외를 통해 모순을 발견하고, AI 또는 룰 기반 판정 엔진이 플레이어의 추론 근거를 평가한다.

## 2. Product Wedge

### 2.1 제품 한 줄 정의

제한된 대화 기회 안에서 용의자의 진술과 증거를 비교해 알리바이의 모순을 찾아내는 AI 보조 추리 게임.

### 2.2 핵심 타깃

| 구분 | 설명 |
| --- | --- |
| 1차 타깃 | 추리 소설, 방탈출, 머더 미스터리, 역전재판류 게임을 좋아하는 15~35세 플레이어 |
| 2차 타깃 | AI 캐릭터 대화형 게임, 인터랙티브 스토리, 논리 퍼즐을 선호하는 사용자 |
| 플레이 상황 | 15~30분 단위의 짧은 싱글 플레이 세션 |
| 구매/사용 동기 | 복잡한 조작보다 논리적 추론과 반전 서사를 즐기고 싶음 |

### 2.3 차별화 포인트

| 차별화 요소 | 설명 |
| --- | --- |
| 알리바이 교차검증 | 플레이어가 질문, 증거, 타인 진술을 직접 연결해 모순을 찾는다. |
| 제한된 대화 횟수 | 대화 자원이 제한되어 어떤 질문을 직접 입력할지 결정하는 과정이 전략이 된다. |
| 증거 기반 AI 판정 | 단순 정답 선택이 아니라, 플레이어가 제시한 근거 조합의 타당성을 평가한다. |
| 한 화면 수사 데스크 | 용의자, 대화, 기록, 증거, 추리 단계가 한 화면에서 연결된다. |
| 사건 제작 확장성 | 사건 데이터를 JSON/DSL로 분리해 새로운 에피소드를 추가할 수 있다. |

### 2.4 제품이 아닌 것

- 자유 채팅만 제공하는 AI 캐릭터 대화 서비스가 아니다.
- 무작위 생성만으로 사건을 만드는 샌드박스가 아니다.
- 액션, 잠입, 전투 중심의 탐정 게임이 아니다.
- 실제 수사, 법률 판단, 심리 분석을 대체하는 도구가 아니다.

## 3. 문제 정의

기존 추리 게임은 정해진 선택지를 순서대로 누르는 구조가 많아 플레이어가 실제로 "추론했다"는 감각이 약해질 수 있다. 본 MVP는 선택지 클릭형 UI가 아니라 자연어 대화형 인터페이스를 기본으로 하되, 내부에서는 사건 그래프, 진술 타임라인, 증거 신뢰도, 모순 규칙을 구조화해 서사 일관성과 판정 안정성을 확보한다.

즉 플레이어에게는 실제 탐문처럼 보이는 자유 입력 대화와 증거 조합을 제공하고, 시스템 내부에서는 CharacterAgent, LightRuleCheck, GameMasterAgent가 순차적으로 대화 생성, 이상 응답 검증, 사건 노트 이벤트 제안을 수행한다. GameMasterAgent는 UI 상태를 직접 바꾸지 않고, Backend Event Processor가 검증한 이벤트를 SSE 또는 WebSocket으로 비동기 발행해 수첩/증거/배경/캐릭터 상태를 업데이트한다.

## 6. Core User Journey

1. 플레이어는 사건 개요를 읽고 용의자 목록과 주요 증거를 확인한다.
2. 남은 대화 횟수 안에서 용의자를 선택하고 자연어 질문을 입력한다.
3. 각 답변은 대화 기록에 저장되며, 중요한 문장은 진술 카드로 추출된다.
4. 플레이어는 증거/기록 탭에서 진술과 충돌하는 단서를 찾는다.
5. 모순 제기 단계에서 "진술 A + 증거 B" 또는 "진술 A + 진술 C"를 연결한다.
6. 판정 엔진은 모순 여부, 근거 충분성, 압박 상승 여부를 평가한다.
7. 압박이 임계값을 넘으면 용의자가 추가 진술 또는 숨긴 사실을 공개한다.
8. 모든 단서를 정리한 뒤 최종 범인을 지목하고, 근거 조합을 제출한다.
9. 엔딩에서 정답, 놓친 단서, 추론 품질, 사용 질문 횟수를 피드백한다.

| F-073 | AI 판정은 룰 엔진 결과를 덮어쓰지 않고 설명 생성과 대화 안정화에 사용한다. | P0 |

## 9. Game Mechanics

### 9.1 질문 횟수

MVP 기본값은 사건당 12회다. 핵심 모순을 맞히거나 숨은 증거를 발견하면 보너스 질문을 1회 지급할 수 있다. 질문 횟수가 0이 되어도 최종 지목은 가능하지만, 추가 정보 수집은 제한된다.

### 9.2 압박 시스템

| 상태 | 조건 | 결과 |
| --- | --- | --- |
| Normal | 기본 상태 | 일반 답변만 제공 |
| Pressed | 관련 모순 1개 이상 적중 | 방어적 답변, 추가 단서 일부 공개 |
| Broken | 핵심 모순 적중 또는 압박 임계값 도달 | 숨긴 진술 공개 |

### 9.3 모순 점수

모순 판정은 다음 요소를 조합한다.

| 요소 | 설명 |
| --- | --- |
| 시간 충돌 | 두 진술 또는 기록의 시간대가 양립 불가능 |
| 위치 충돌 | 같은 시간에 다른 장소에 있었다는 증거 |
| 관계 충돌 | 부인한 관계가 다른 증거로 드러남 |
| 물증 충돌 | 증거의 상태가 진술과 맞지 않음 |
| 신뢰도 | CCTV, 출입기록, 목격담 등 출처별 신뢰도 |

### 9.4 판정 안정화

PDF의 K-of-N voting 개념을 차용해, AI/룰 판정은 단일 키워드 매칭으로 확정하지 않는다. 예를 들어 핵심 모순 판정은 최소 2개 이상의 조건이 충족되어야 한다.

| 판정 | 예시 조건 |
| --- | --- |
| 정답 | 필수 진술 1개 + 필수 증거 1개 + 같은 타임라인 구간 |
| 부분 정답 | 필수 진술 또는 필수 증거 중 하나만 포함 |
| 근거 부족 | 관련 인물은 맞지만 연결 근거가 약함 |
| 오답 | 사건 그래프상 관련 없는 조합 |

## 12. Technical Architecture

### 12.1 MVP 구조

| 레이어 | 권장 스택 | 역할 |
| --- | --- |
| Frontend | Web FE | 게임 화면, 상태 표시, 대화/증거 인터랙션 |
| Backend API | FastAPI | 세션, 사건 진행 상태, 사용자 입력, 저장/불러오기, 룰 엔진 호출 |
| AI Service | FastAPI + LangGraph | CharacterAgent, LightRuleCheck, GameMasterAgent 기반 대화 생성/검증/이벤트 제안 워크플로우 오케스트레이션 |
| Case Engine | Backend 내부 모듈 | 사건 데이터 로딩, 질문 제한, 해금 조건 관리 |
| Rule Engine | Backend 내부 모듈 | 모순 판정, 압박 수치, 최종 정답 판정 |
| Persistence | DB 또는 파일 저장소 | 진행 상태, 대화 기록, 발견 단서, 사건 데이터 저장 |

### 12.2 서비스 분리 기준

MVP는 `Frontend`, `Backend API`, `AI Service`의 3개 실행 단위로 구성한다. Backend API와 AI Service는 모두 FastAPI로 구성하되 책임을 분리한다.

| 서비스 | 책임 | 외부 공개 여부 |
| --- | --- | --- |
| Frontend | 플레이 화면, 상태 렌더링, 사용자 인터랙션 | 사용자에게 공개 |
| Backend API | 게임 상태의 단일 진실 공급원, 룰 기반 판정, 저장/조회 API | Frontend에 공개 |
| AI Service | LangGraph 기반 AI 플로우 실행, LLM 호출, 자연어 생성 | Backend API에서만 호출 |

AI Service는 게임 정답을 직접 확정하지 않는다. 정답성, 해금 조건, 대화 횟수, 압박 수치 변경은 Backend API의 Rule Engine과 Event Processor가 결정한다. AI Service는 Backend가 전달한 사건 상태와 판정 결과를 바탕으로 CharacterAgent가 캐릭터 답변을 만들고, LightRuleCheck가 이상 응답을 검증하며, GameMasterAgent가 대화에서 드러난 정보를 `proposedEvents[]`로 반환한다. Backend만 이벤트를 적용하고 SSE/WebSocket으로 UI에 발행한다.

### 12.3 판정 우선순위

1. 룰 엔진이 사건 상태와 제출 근거를 결정적으로 판정한다.
2. AI는 판정 결과를 자연어 설명으로 바꾼다.
3. AI 출력이 룰 엔진 판정과 충돌하면 룰 엔진 결과를 우선한다.
4. AI 실패 시에도 룰 기반 게임 플레이는 계속 가능해야 한다.

### 12.4 API 흐름

| 흐름 | 설명 |
| --- | --- |
| 게임 시작 | FE -> BE: 사건 ID로 새 세션 생성 |
| 대화 입력 | FE -> BE: 용의자 ID와 플레이어 질문 텍스트 제출, BE가 상태 갱신 |
| 답변 생성/검증/이벤트 제안 | BE -> AI Service: 사건 상태, 캐릭터 상태, 허용 진술 범위, 현재 긴장도를 전달. CharacterAgent → LightRuleCheck → GameMasterAgent 순서로 처리하고 proposedEvents 반환 |
| 비동기 UI 업데이트 | BE -> FE: Event Processor가 검증한 NOTE/EVIDENCE/TIMELINE/TENSION/VISUAL 이벤트를 SSE 또는 WebSocket으로 발행 |
| 모순 제기 | FE -> BE: 진술/증거 조합 제출, BE Rule Engine이 판정 |
| 힌트/노트 요약 | FE -> BE -> AI Service: 현재까지 발견한 기록 기반으로 요약 생성 |
| 최종 지목 | FE -> BE: 범인과 근거 제출, BE가 최종 판정과 엔딩 데이터 반환 |

### 12.5 이벤트 기반 상태 업데이트

GameMasterAgent가 만든 상태 변경 후보는 동기 HTTP 응답만으로 UI에 직접 반영하지 않는다. Backend Event Processor가 `proposedEvents[]`를 검증해 event store에 저장하고, `GET /api/v1/sessions/{sessionId}/events` SSE 또는 WebSocket 채널로 발행한다. MVP는 단방향 서버 이벤트만 필요하므로 SSE를 우선한다. FE는 마지막으로 처리한 `eventId`를 저장해 재연결 시 누락된 수첩/증거/visualState 업데이트를 복구한다.

### 12.6 프론트엔드 스택 결정 보류

디자인이 확정되기 전까지 프론트엔드는 "Web FE"로 정의한다. 다만 공유 UI 이미지처럼 정보 밀도가 높은 단일 화면 게임이므로, 실제 구현 시에는 컴포넌트 기반 SPA 구조가 적합하다. 디자인 확정 후 React, Next.js, Vite 중 하나를 선택한다.

### 12.7 로컬 우선 설계

PDF의 온디바이스 설계 관점을 제품에 맞게 적용한다. 핵심 게임 판정은 네트워크나 외부 모델에 의존하지 않고 로컬에서 실행한다. 외부 AI 호출은 사용자 경험을 풍부하게 만드는 보조 기능으로 제한한다.

## 13. Metrics

### 13.1 North Star Metric

| 지표 | 정의 |
| --- | --- |
| Valid Case Completion Rate | 사건을 시작한 플레이어 중 최종 지목까지 도달하고, 최소 1개 이상의 핵심 모순을 발견한 비율 |

### 13.2 제품 지표

| 지표 | 목적 |
| --- | --- |
| Case Completion Rate | 사건 완주 가능성과 몰입도 측정 |
| Core Contradiction Discovery Rate | 핵심 추리 루프가 작동하는지 측정 |
| Average Questions Used | 질문 제한 난이도 조정 |
| Wrong Accusation Rate | 최종 판정 난이도 조정 |
| Hint Usage Rate | 단서 배치와 난이도 검증 |
| Replay Rate | 사건 구조와 분기 재방문 가치 측정 |

### 13.3 판정 품질 지표

PDF의 Precision/Recall 균형 관점을 차용한다.

| 지표 | 설명 |
| --- | --- |
| Contradiction Precision | 정답으로 판정한 모순 중 실제 정답 규칙과 일치한 비율 |
| Contradiction Recall | 실제 핵심 모순 중 플레이어 제출 시 정답으로 인정된 비율 |
| False Positive Review Count | 틀린 근거를 맞다고 처리한 사례 수 |
| False Negative Review Count | 맞는 근거를 놓친 사례 수 |
| MinPR | 주요 판정 Precision/Recall 중 최저값. 판정 엔진의 보수적 품질 지표 |

| AC-001 | 플레이어가 새 게임을 시작해 최종 범인 지목까지 중단 없이 진행할 수 있다. |
| AC-002 | 질문을 입력해 전송하면 남은 대화 횟수가 정확히 감소하고 대화 기록에 저장된다. |
| AC-003 | 핵심 진술과 핵심 증거를 제출하면 정답 모순으로 판정된다. |
| AC-004 | 관련 없는 조합을 제출하면 오답 또는 근거 부족으로 판정된다. |
| AC-005 | 핵심 모순 정답 시 용의자 상태가 변하고 추가 진술 또는 증거가 해금된다. |
| AC-006 | 최종 범인 지목은 범인 ID와 필수 근거 조합을 함께 평가한다. |
| AC-007 | 새로고침 후에도 진행 상태를 복구할 수 있다. |
| AC-008 | AI 기능이 실패해도 룰 기반 사건 진행은 막히지 않는다. |

## 16. Risks and Mitigations

| 리스크 | 영향 | 대응 |
| --- | --- | --- |
| AI가 사건 설정과 다른 답변 생성 | 추리 공정성 훼손 | LightRuleCheck로 이상 응답을 검증하고, 룰 엔진 결과 우선, AI는 승인된 진술 노드 내에서만 답변 |
| GameMasterAgent가 잘못된 상태 변경을 제안 | 수첩/증거/타임라인 오염 | GameMasterAgent는 직접 상태를 변경하지 않고 proposedEvents만 반환. Backend Event Processor가 스키마, 공개 범위, 해금 조건을 검증한 뒤에만 저장/발행 |
| 정답 조합이 너무 좁음 | 플레이어가 맞는 추론을 해도 오답 처리 | 부분 정답과 동의어/대체 증거 규칙 지원 |
| UI 정보량 과다 | 플레이어가 단서를 놓침 | 탭 구조, 중요 단서 배지, 추리 노트 제공 |
| 질문 제한이 과도함 | 중도 이탈 증가 | 난이도별 질문 횟수, 보너스 질문, 힌트 도입 |
| 사건 추가 비용 큼 | 콘텐츠 확장 어려움 | 사건 데이터 스키마와 검증 도구 우선 구축 |
