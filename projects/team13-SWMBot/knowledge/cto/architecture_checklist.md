# AI 서비스 아키텍처 체크리스트

## 개요
AI 서비스는 일반 웹 서비스와 달리 비용 폭발, 확률적 오류, 외부 API 의존이라는 세 가지 고유 리스크를 가진다. MVP 단계부터 비용 제어 레이어, 장애 처리, 관측성(Observability)을 설계하지 않으면 스케일 시점에 전면 재설계가 불가피하다.

---

## AI 서비스 필수 아키텍처 구성요소

### 전체 구조 다이어그램

```
클라이언트(앱/웹)
        │
        ▼
[API Gateway / Rate Limiter]   ← 비용 제어 1차 관문
        │
        ▼
[Application Server]
        │
   ┌────┴────────────┐
   ▼                 ▼
[캐시 레이어]    [AI 오케스트레이션 레이어]
(Redis/CDN)      (LangChain, 자체 로직)
                       │
              ┌────────┼────────┐
              ▼        ▼        ▼
         [LLM API] [STT API] [Vector DB]
         (OpenAI)  (Whisper)  (Pinecone)
                       │
                       ▼
              [모니터링/로깅 레이어]
              (LangSmith, Datadog)
```

### 비용 제어 레이어 필수 항목

| 제어 항목 | 구현 방법 | 미구현 시 리스크 |
|---------|---------|--------------|
| 사용자별 Rate Limit | Redis로 슬라이딩 윈도우 카운터 | 1명이 무한 호출로 비용 폭발 |
| 일별/월별 사용량 캡 | DB에 usage 테이블 + 미들웨어 체크 | 무료 체험 남용으로 수천만 원 청구 |
| 토큰 길이 제한 | 입력 전처리에서 max_tokens 강제 | 긴 입력으로 예산 초과 |
| 응답 캐싱 | 동일 입력 해시 → Redis 캐시 조회 | 동일 질문 반복 시 불필요한 API 호출 |
| 비용 알림 | AWS Cost Explorer, Stripe webhook | 예산 초과를 뒤늦게 발견 |

## 장애 유형별 처리 방법

| 장애 유형 | 감지 방법 | 처리 전략 | 사용자 표시 |
|---------|---------|---------|-----------|
| LLM API 타임아웃 | 30초 후 TimeoutError | 재시도 1회 후 캐시/Fallback 응답 | "잠시 후 다시 시도해 주세요" |
| LLM API 5xx 오류 | HTTP 상태 코드 | Exponential backoff 재시도 (3회) | "AI 서비스가 일시적으로 불안정합니다" |
| LLM API 완전 중단 | 헬스체크 실패 | 예비 모델로 전환 (GPT-4 → Claude) | "임시 모드로 운영 중입니다" |
| 환각 감지 | 출력 후처리 규칙 | Human-review 큐에 삽입 | 낮은 신뢰도 뱃지 표시 |
| 벡터 DB 오류 | 쿼리 실패 | LLM 단독 응답으로 fallback | 사용자에게 불표시 |
| 토큰 한도 초과 | 429 Too Many Requests | 큐에 넣고 지연 처리 | "처리 대기 중입니다" |
| 컨텍스트 길이 초과 | 4096/8192/128k 토큰 한도 | 입력 청킹(chunking) | 자동 처리, 표시 불필요 |

### Exponential Backoff 패턴

```python
import time
import random

def call_llm_with_retry(prompt, max_retries=3):
    for attempt in range(max_retries):
        try:
            return client.chat.completions.create(...)
        except openai.RateLimitError:
            if attempt == max_retries - 1:
                raise
            wait = (2 ** attempt) + random.uniform(0, 1)
            time.sleep(wait)  # 1초, 2초, 4초 + 지터
```

| 지표 | 측정 도구 | 경고 임계값 예시 |
|------|---------|--------------|
| LLM API 응답 시간 (P95) | LangSmith, Datadog | P95 > 10초 |
| 환각/오류 비율 | 자체 평가 파이프라인 | > 5% |
| 일별 LLM 비용 | OpenAI Dashboard | 예산의 80% 도달 시 알림 |
| API 오류율 | Sentry, Datadog | > 1% |
| 캐시 히트율 | Redis Metrics | < 30% (최적화 검토) |
| 사용자별 토큰 사용량 | 자체 DB | 이상치 탐지 |

## 체크리스트

- [ ] 사용자별 API 호출 횟수 제한(Rate Limit)이 구현되어 있는가?
- [ ] 무료 체험 사용자의 총 사용량 캡이 정의되고 구현되어 있는가?
- [ ] LLM API 타임아웃/오류 시 사용자에게 표시할 메시지가 설계되어 있는가?
- [ ] 예비 모델(Fallback LLM)이 지정되어 있는가?
- [ ] LLM 호출 비용을 사용자·기능·날짜별로 추적하는가?
- [ ] 비용 초과 시 알림 체계(Slack, 이메일 등)가 있는가?
- [ ] 프롬프트와 응답을 로깅하는가? (디버깅, 품질 모니터링 목적)
- [ ] 동일 입력에 대한 응답 캐싱 전략이 있는가?
- [ ] 개인정보가 포함된 입력에 대한 처리 방침이 있는가?

## 흔한 레드플래그

- **사용량 제한 없는 무료 체험**: "가입하면 무료로 써보세요" + Rate Limit 없음 = 런칭 첫날 수백만 원 청구 가능
- **개인정보를 그대로 프롬프트에 포함**: 사용자 이름, 이메일, 주민번호 등이 LLM API로 전송되는 구조
- **단일 모델 의존**: OpenAI 또는 Anthropic 장애 = 서비스 전체 다운. LLM 공급자 장애는 연 수십 회 발생하므로 멀티 모델 폴백 전략 필수
- **로깅 없음**: LLM 응답 품질을 측정할 방법이 없어 개선 불가
- **토큰 길이 무제한**: 사용자가 긴 문서를 붙여넣으면 한 번에 수천 원짜리 API 호출 발생
- **동기 처리만 있음**: 오래 걸리는 AI 작업을 HTTP 요청에 직접 물리면 타임아웃 빈발
