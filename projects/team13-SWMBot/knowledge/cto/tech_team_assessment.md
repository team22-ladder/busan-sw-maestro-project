# 기술 팀 역량 평가 및 실행력 검증

## 개요
아이디어와 기술 계획이 아무리 훌륭해도 팀이 실행할 역량이 없으면 무의미하다. CTO는 팀 규모, 기술 역량, 외주 의존도를 기반으로 제시된 로드맵이 실현 가능한지를 냉정하게 평가해야 한다. AI 스타트업에서 기술 팀의 질은 서비스 성패와 직결된다.

## 기술 역량 인터뷰 질문 목록

### LLM 엔지니어링 검증 질문

```
기초 수준 확인:
- "GPT-4o와 GPT-4o-mini의 비용 차이와 선택 기준을 설명해 주세요."
- "RAG를 구현할 때 청킹(chunking) 전략을 어떻게 결정합니까?"
- "LLM 응답의 품질을 자동으로 평가하는 방법은 무엇입니까?"

중급 수준 확인:
- "벡터 검색과 키워드 검색을 언제 어떻게 조합합니까?"
- "LLM 비용을 50% 줄이기 위한 전략을 3가지 제시해 주세요."
- "Prompt Injection 공격 예시를 들고 방어 방법을 설명해 주세요."

고급 수준 확인:
- "파인튜닝 vs 프롬프트 엔지니어링 vs RAG를 언제 선택합니까?"
- "멀티 에이전트 시스템에서 루프 방지와 오류 복구를 어떻게 구현합니까?"
- "모델 드리프트(Model Drift)를 어떻게 감지하고 대응합니까?"
```

## 외주·파트너십 의존 리스크 분석

| 의존 유형 | 허용 범위 | 불허 범위 | 리스크 |
|---------|---------|---------|------|
| LLM API (OpenAI, Anthropic) | 허용 (단, 폴백 모델 준비) | 단일 API 완전 의존 (폴백 없음) | 서비스 중단, 가격 인상 |
| 클라우드 서비스 (AWS 등) | 허용 (표준적) | 특정 벤더 독점 기능 과도 사용 | 마이그레이션 비용 |
| AI 기능 전부 외주 | 불허 | - | 핵심 기술력 부재, 종속 |
| UI/UX 디자인 외주 | 허용 | - | 낮음 |
| 어드민/대시보드 외주 | 부분 허용 | 핵심 AI 로직까지 외주 | 내부화 어려움 |
| 데이터 어노테이션 외주 | 허용 | 품질 검수 없는 완전 위탁 | 데이터 품질 저하 |

## 팀 구성 현실성 평가 매트릭스

```
[팀 구성 평가 예시]

발표: "저희 팀은 2인 팀으로 6개월 안에 iOS 앱, Android 앱,
      AI 튜터 기능, 어드민 대시보드, 결제 시스템을 완성합니다."

계산:
  기능별 최소 공수(man-week):
  - iOS 앱 (기본): 8주
  - Android 앱 (기본): 8주
  - AI 튜터 파이프라인: 6주
  - 어드민 대시보드: 4주
  - 결제 시스템 연동: 2주
  - 인프라/배포: 2주
  합계: 30주 (단일 개발자 기준)

  2인 팀 × 6개월(24주) = 48 man-week
  → 요구 공수 30주가 48 man-week 내 가능해 보이나,
     PM·기획 변경·버그 수정·회의 등 30~40% 오버헤드 적용 시
     실질 개발 공수: 48 × 0.65 = 31 man-week → 매우 타이트

  판단: iOS 또는 Android 중 하나 제거 권고
```

## 흔한 레드플래그

- **2인 팀이 앱+AI+어드민+대시보드 6개월 계획**: 각 기능의 공수를 계산하면 물리적으로 불가능
- **"AI 개발 전부 외주"**: 핵심 기술력이 팀에 없으면 장기적으로 외주에 종속
- **기술 리더의 LLM 실무 경험 없음**: 프롬프트 몇 번 써본 것을 AI 개발 경험이라고 주장
- **"개발은 나중에 합류할 팀원이 담당"**: 핵심 기술 인력 채용 전에 투자받으려는 구조
- **역할 없는 팀 구성**: "팀원 전부 AI 엔지니어" = 프론트엔드·백엔드·인프라 담당 없음
- **기술 스택 나열만 있고 아키텍처 없음**: "React, Python, LangChain 씁니다" 수준의 답변
- **오픈소스 프로젝트 없음**: AI 스타트업 CTO가 공개 포트폴리오(GitHub) 전무
