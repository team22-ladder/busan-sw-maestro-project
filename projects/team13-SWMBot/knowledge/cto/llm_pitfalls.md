# LLM 적합성 판단 및 주요 함정

## 개요
LLM은 강력하지만 모든 문제에 적합하지 않다. CTO는 LLM이 정말 필요한지, 환각을 어떻게 처리할지, 응답 지연을 어떻게 감당할지를 구체적으로 검증해야 한다. 과도한 LLM 의존은 비용 폭발, UX 저하, 신뢰성 문제로 이어진다.

---

## LLM 적합 vs 비적합 판단 기준

| 기준 | LLM 적합 | LLM 불필요 |
|------|----------|------------|
| 입력 형태 | 자연어, 비정형 텍스트 | 정형 데이터, 숫자 연산 |
| 출력 형태 | 자유형 생성, 요약, 번역 | 분류(클래스 고정), 정렬, 검색 |
| 정밀도 요구 | 80~90% 허용 가능 | 99.9% 이상 요구 (금융, 의료) |
| 실시간성 | 1~5초 허용 | 100ms 이하 필수 |
| 비용 민감도 | 요청당 $0.01 허용 | 요청당 $0.0001 이하 필요 |
| 예시 사례 | 코드 리뷰, 문서 요약, 챗봇 | 로그인 검증, 결제 계산, CRUD |

**판단 규칙**: 규칙 기반(Rule-based)으로 해결 가능한 문제에 LLM을 쓰는 것은 과도한 비용 낭비다. LLM은 "규칙을 정의하기 어려운" 비정형 문제에만 써야 한다.

## 환각(Hallucination) 처리 전략 비교

| 전략 | 설명 | 정확도 개선 | 비용 | 지연 | 적합 케이스 |
|------|------|------------|------|------|------------|
| RAG (Retrieval-Augmented Generation) | 외부 DB에서 근거 문서 검색 후 프롬프트에 삽입 | 높음 | 중간 | +0.5~1초 | 사실 기반 Q&A, 문서 검색 |
| Self-consistency | 동일 질문 3~5회 반복 후 다수결 | 중간 | 3~5배 비용 | 3~5배 지연 | 수학 추론, 코드 생성 |
| Confidence Threshold | 모델 로그 확률(logprob) 기반 신뢰도 필터링 | 낮음~중간 | 낮음 | 없음 | 분류, 감정 분석 |
| Human-in-the-loop | 낮은 신뢰도 답변을 사람이 검토 | 매우 높음 | 인건비 | 수분~수시간 | 의료, 법률, 금융 |
| Constitutional AI / Guardrails | 출력 후처리 규칙으로 검증 | 중간 | 낮음 | +0.1~0.3초 | 콘텐츠 안전, 형식 준수 |

### CTO 핵심 검증 질문
> "환각 발생 시 사용자에게 어떻게 표시합니까? '모르겠습니다' 응답과 잘못된 정보를 어떻게 구분합니까?"

- RAG를 쓴다면: 검색 소스를 UI에 표시하는가? (예: "출처: 공식 문서 3페이지")
- Confidence Threshold를 쓴다면: 임계값은 얼마이고, 그 이하일 때 fallback 처리는?
- Human-in-loop이라면: 검토 SLA(Service Level Agreement)는 몇 시간인가?

```
TTFT (Time to First Token):
- GPT-4o-mini:      200~400ms  | 80~100 tokens/sec
- GPT-4o:           ~464ms     | —
- Claude Haiku 4.5: 500~639ms  | 80~120 tokens/sec
- Claude Sonnet 4.6:500~800ms  | ~53 tokens/sec
- Gemini 2.5 Flash: ~620ms     | 146~173 tokens/sec
- Claude Opus 4.8:  1,500~2,000ms | 45~66 tokens/sec
- o3 (추론 모델):   2~150초    | — (체인오브소트 처리)
- o4-mini:          1~30초     | — (경량 추론)

※ 추론 모델(o3, o4-mini)은 답변 생성 전 내부 사고 과정으로 TTFT가 매우 길어 실시간 채팅 UX에 부적합
```

### Streaming 대응 필수 여부
- **스트리밍 필수**: 사용자가 응답을 기다리는 실시간 채팅, 작문 보조
- **스트리밍 불필요**: 배치 분석, 비동기 보고서 생성
- Streaming 구현 시 주의: 중간에 연결 끊김 처리, 불완전한 JSON 파싱 오류

```python
# Streaming 기본 패턴 (OpenAI)
for chunk in client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    stream=True
):
    if chunk.choices[0].delta.content:
        yield chunk.choices[0].delta.content
        # 주의: 연결 끊김 시 재시도 로직 필요
```

## 흔한 레드플래그

- **"모든 기능에 Claude Opus 4.8"**: Haiku 4.5 대비 5배 비용. 간단한 분류·요약에 최고가 모델을 쓰는 것은 비용 낭비
- **폴백(Fallback) 없음**: LLM API가 5xx 오류를 반환할 때 서비스 전체가 다운되는 구조
- **환각 오류를 무시**: "가끔 틀리는 건 어쩔 수 없어요"라는 답변. 특히 의료·법률·금융 도메인에서 치명적
- **지연 과소평가**: "AI가 바로바로 답해줍니다"라고 하면서 GPT-4 응답 시간 고려 없음
- **프롬프트 엔지니어링 = AI 개발**: 프롬프트만 바꿔서 서비스를 만들었다고 표현. 실제 평가 파이프라인, 장애 처리, 비용 관리가 없음
- **Self-consistency를 무분별하게 적용**: 비용 3~5배, 지연 3~5배가 되는 것을 인지하지 못함
