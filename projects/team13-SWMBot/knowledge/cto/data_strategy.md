# AI 서비스 데이터 전략 및 데이터 해자

## 개요
AI 서비스의 장기 경쟁력은 모델보다 데이터에서 나온다. 누구나 같은 LLM API를 쓸 수 있지만, 고품질 도메인 특화 데이터는 모방이 어렵다. CTO는 데이터 수집·처리·활용 전략이 실현 가능한지, 법적 리스크는 없는지를 검증해야 한다.

---

## 학습 데이터 종류별 특성

| 데이터 종류 | 수집 방법 | 품질 | 비용 | 법적 리스크 | 경쟁 차별화 |
|-----------|---------|------|------|-----------|-----------|
| 공개 데이터셋 | Hugging Face, 공공데이터포털 | 중간 | 낮음 | 낮음 | 낮음 (누구나 가능) |
| 웹 크롤링 | BeautifulSoup, Scrapy | 불균일 | 중간 | 중간~높음 | 낮음 |
| 사용자 생성 데이터 | 서비스 내 로그, 피드백 | 높음 | 낮음 (운영비) | 동의 필수 | 매우 높음 |
| 어노테이션 데이터 | Upstage, Scale AI, 크라우드소싱 | 매우 높음 | 높음 | 낮음 | 높음 |
| 파트너십 데이터 | B2B 계약, 데이터 거래소 | 높음 | 중간~높음 | 계약 의존 | 높음 |
| 합성 데이터 (Synthetic) | LLM으로 생성 | 중간~높음 | 낮음 | 낮음 | 중간 |

### 각 데이터 유형별 주의사항

**공개 데이터셋**
- 라이선스 확인 필수: CC-BY, CC-BY-SA, GPL 등 상업적 이용 가능 여부
- 한국어 공개 데이터: AI Hub (aihub.or.kr), 국립국어원, KLUE 벤치마크

**웹 크롤링**
- robots.txt 준수 여부 확인
- 저작권법 위반 여부 (뉴스, 블로그, 도서 등)
- OpenAI vs NYT 소송 참고 (2023): 크롤링 데이터의 저작권 분쟁 현재진행형

**사용자 생성 데이터**
- 개인정보보호법 제15조: 수집 목적 명시 및 동의 필수
- 서비스 약관에 "데이터 활용" 조항 포함 여부
- GDPR 적용 대상(EU 사용자)이면 별도 동의 메커니즘

## 데이터 해자(Data Moat) 선순환 구조

```
        사용자 유입
             │
             ▼
    ┌─── 서비스 사용 ────────┐
    │         │             │
    │         ▼             │
    │   사용자 데이터 수집    │
    │    (로그, 피드백)      │
    │         │             │
    │         ▼             │
    │   모델/프롬프트 개선    │
    │         │             │
    │         ▼             │
    │   서비스 품질 향상 ─────┘
    │         │
    │         ▼
    └──► 더 많은 사용자 유입

데이터 해자 형성 조건:
1. 사용자가 직접 생성하거나 행동하는 고품질 데이터
2. 타사가 쉽게 복제하기 어려운 도메인 특화 데이터
3. 양이 아닌 품질 + 다양성이 핵심
```

### 데이터 해자 형성 기간 현실

```
데이터 유형별 의미 있는 규모 도달 예상 기간:
- 사용자 행동 로그: MAU 1천 명 도달 시 3~6개월
- 도메인 특화 어노테이션: 1,000건 기준 1~3개월 (예산 $5,000~$20,000)
- 파인튜닝용 고품질 데이터: 최소 1,000~10,000건 (도메인마다 다름)
- 경쟁사가 복제 불가한 수준: 최소 2~3년 누적 필요
```

## 개인정보 처리 규정 핵심

### 국내 규정 (개인정보보호법)

| 항목 | 요구사항 | AI 서비스 적용 포인트 |
|------|---------|---------------------|
| 수집 동의 | 수집 목적, 항목, 보유 기간 고지 후 동의 | 회원가입 시 약관 + 개인정보 처리방침 |
| 최소 수집 원칙 | 서비스에 필요한 최소한만 수집 | 불필요한 사용자 정보를 LLM에 전달 금지 |
| 보유 기간 제한 | 목적 달성 후 즉시 파기 | 대화 로그 보관 기간 명시 필요 |
| 제3자 제공 | 별도 동의 없이 외부 전송 금지 | LLM API 전송 = 제3자 제공 해당 가능성 |
| 열람/삭제권 | 사용자 요청 시 데이터 열람·삭제 | 대화 내역 삭제 기능 구현 필요 |

### LLM API 전송 시 주의사항

```
주의가 필요한 개인정보 유형:
- 이름, 이메일, 전화번호 → 익명화 후 전송 또는 전송 금지
- 주민등록번호, 신용카드 번호 → 절대 전송 금지
- 의료 정보, 정치적 의견 → 민감정보로 별도 동의 필요
- IP 주소 → 개인 식별 가능성 있음

안전한 처리 방법:
- 마스킹 처리: "홍길동(010-1234-5678)" → "사용자A(***-****-****)"
- 익명화: 개인 식별 정보 제거 후 LLM 전송
- 데이터 처리 계약(DPA): OpenAI/Anthropic와 별도 계약 체결 여부 확인
```

## 데이터 품질 관리

| 품질 지표 | 측정 방법 | 기준값 예시 |
|---------|---------|-----------|
| 데이터 정확성 | 샘플링 후 수동 검수 | 95% 이상 |
| 레이블 일관성 | Inter-Annotator Agreement (Cohen's Kappa) | κ > 0.8 |
| 클래스 불균형 | 클래스별 분포 확인 | 비율 1:10 이하 |
| 중복 데이터 | 해시 또는 유사도 기반 중복 제거 | 중복률 5% 이하 |
| 데이터 신선도 | 생성 날짜 추적 | 도메인별 기준 다름 |

## 흔한 레드플래그

- **동의 없는 학습 데이터 수집**: 사용자 대화를 "서비스 개선 목적"으로 동의 없이 학습에 활용
- **저작권 미검토 크롤링**: 뉴스, 소설, 교재 등 저작권 있는 콘텐츠를 무단 크롤링
- **"데이터는 나중에 모으면 됩니다"**: 데이터 수집 전략 없이 모델부터 만들려는 접근
- **합성 데이터 과신**: LLM으로 생성한 합성 데이터만으로 실제 도메인 성능 달성 불가
- **어노테이션 비용 과소평가**: 1,000건 어노테이션에 $5,000~$20,000 이상 소요 가능
- **데이터 파이프라인 부재**: 지속적인 데이터 수집·정제·버저닝 체계 없이 일회성 수집만 계획
