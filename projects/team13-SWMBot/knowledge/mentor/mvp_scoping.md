# MVP 범위 설정 (MVP Scoping)

## 개요
MVP(Minimum Viable Product)의 목적은 "기능을 완성하는 것"이 아니라 "핵심 가설을 가장 빠르고 저렴하게 검증하는 것"이다. AI 스타트업에서 가장 흔한 실수는 완성형 제품에 가까운 것을 MVP라고 부르며 6개월을 소비하는 것이다.

---

## 1. MVP의 핵심 목적: 가설 검증

MVP를 만들기 전에 반드시 다음 형식으로 핵심 가설을 1문장으로 정의해야 한다.

```
핵심 가설 템플릿:
"[타겟 고객]은 [핵심 기능/가치]에 대해 [성공 지표]를 달성할 것이다."

예시:
- "직장인 영어 학습자는 AI 발음 교정 서비스에 대해 첫 사용 후 7일 리텐션 30%를 달성할 것이다."
- "중소 쇼핑몰 운영자는 AI 상품 설명 자동 생성 기능에 대해 월 2만원 지불 의향을 보일 것이다."
```

MVP가 이 가설을 검증하는 데 필요한 기능만 포함해야 한다.

---

## 2. 기능 포함 여부 판단 기준

```
기능 포함 결정 트리:

[새 기능 후보]
      │
      ▼
이 기능 없이는 핵심 가설을 검증할 수 없는가?
      │
   YES ─────────────────────── 포함
      │
      NO
      │
      ▼
이 기능이 없으면 고객이 핵심 가치를 경험할 수 없는가?
      │
   YES ─────────────────────── 포함 (단, 가장 단순한 형태로)
      │
      NO
      │
      ▼
      제외 (백로그에 추가)
```

### 판단 예시

| 기능 | 핵심 가설과의 관계 | 판단 |
|---|---|---|
| AI 발음 교정 (핵심 기능) | 가설 직접 검증 | 포함 |
| 학습 진도 대시보드 | 가설 검증에 불필요 | 제외 |
| 소셜 랭킹/친구 초대 | 가설 검증에 불필요 | 제외 |
| 회원가입/로그인 | 없으면 사용 불가 | 최소 형태로 포함 |
| 결제 시스템 | 지불 의향 가설이면 포함 | 조건부 포함 |
| 어드민 대시보드 | 운영 편의, 검증 불필요 | 제외 |
| 다국어 지원 | 검증 단계에서 불필요 | 제외 |

---

## 3. AI 스타트업 MVP에서 버려야 할 것들

대부분의 팀이 "나중에 필요할 것"이라며 MVP에 포함시키지만, 실제로는 검증을 지연시키는 것들:

| 제거 대상 | 이유 |
|---|---|
| 소셜 기능 (친구, 랭킹, 공유) | PMF 확인 전 바이럴 설계는 낭비 |
| 어드민/운영 대시보드 | 초기엔 수동 처리로 충분 |
| 알림 시스템 (푸시/이메일) | 리텐션 개선은 PMF 이후 |
| 다국어·현지화 | 한 시장에서 먼저 검증 |
| 상세 통계·분석 화면 | 팀 내부 DB 쿼리로 대체 가능 |
| AI 모델 파인튜닝 | 범용 API로 먼저 가설 검증 후 결정 |
| 고급 개인화 추천 | 기본 기능 retention 확인 후 |
| 오프라인 모드 | 초기 사용자는 온라인 환경 제어 가능 |

> **컨시어지 MVP 원칙:** 자동화가 필요한 부분을 사람이 수동으로 처리해도 고객 가치를 전달할 수 있다면, 자동화는 검증 이후로 미뤄라.

---

## 4. MVP 단계별 로드맵

```
단계 1: 컨시어지 MVP (1~4주)
├─ 목적: 고객이 가치를 경험하는지 확인
├─ 방법: 수동 처리 + 기초 UI (Figma, 노코드)
├─ 성공 기준: 10명이 "이게 진짜 필요하다"고 반복 사용
└─ 비용: 최소화, 팀 공수만 투입

단계 2: 핵심 기능 MVP (4~12주)
├─ 목적: 핵심 가설 수치 검증
├─ 방법: 핵심 기능 1~3개만 구현, 나머지 수동/미구현
├─ 성공 기준: D7 리텐션 25%, 지불 의향 전환률 5%+
└─ 비용: 엔지니어링 최소 투입

단계 3: 완성형 제품 (PMF 확인 후)
├─ 목적: 검증된 가설 기반 스케일링
├─ 방법: 전체 기능 개발, 자동화, 성능 최적화
└─ 성공 기준: MoM 성장률 20%+, NPS 40+
```

---

## 5. 성공 기준 정의

MVP를 시작하기 전에 "어떤 수치가 나오면 가설이 검증된 것인가"를 반드시 정의해야 한다.

| 단계 | 지표 | 최소 기준 예시 |
|---|---|---|
| 문제 검증 | 인터뷰에서 "꼭 필요하다" 응답률 | 20명 중 12명 이상 (60%) |
| 제품 검증 | D7 리텐션 | 25% 이상 |
| 가치 검증 | Sean Ellis 테스트 ("없으면 매우 아쉽다") | 응답자 40% 이상 |
| 수익 검증 | 지불 의향 전환률 | 무료 사용자의 5% 이상 |
| 성장 검증 | MoM 사용자 증가율 | 20% 이상 (3개월 연속) |

## 체크리스트

- [ ] 핵심 가설이 "타겟 고객 + 핵심 기능 + 성공 지표" 형식으로 1문장으로 정의되어 있는가?
- [ ] MVP에 포함된 각 기능이 핵심 가설 검증에 필요한 이유가 설명되어 있는가?
- [ ] MVP 완료 시점의 성공/실패 기준이 수치로 정의되어 있는가?
- [ ] 소셜 기능, 어드민, 알림 등 "나중에 필요한 것들"이 제거되어 있는가?
- [ ] 컨시어지 단계로 먼저 가치 검증을 할 수 있는지 검토했는가?
- [ ] AI 모델 파인튜닝 없이 범용 API로 먼저 검증 가능한지 확인했는가?
- [ ] MVP 개발 기간이 12주 이내로 설정되어 있는가?

---

## 흔한 레드플래그

- **기능이 10개 이상인 MVP**: MVP는 Minimum이다. 10개 기능이 동시에 필요한 가설은 존재하지 않는다.
- **성공 기준이 없다**: "출시해보고 반응을 보겠다"는 것은 검증이 아닌 출시다.
- **"MVP니까 퀄리티는 낮아도 된다"**: MVP는 작지만 핵심 가치는 완전히 전달해야 한다.
- **AI 모델 완성도를 MVP 조건으로 건다**: "모델 정확도가 90% 되면 출시"는 검증을 무기한 지연시킨다.
- **팀 모두가 동의하는 기능 목록 = MVP**: 팀 합의는 고객 검증과 다르다.
- **"경쟁사가 다 있으니 우리도 있어야 한다"**: 경쟁사 기능 벤치마킹은 MVP 스코핑 원칙이 아니다.
