# 피벗 시나리오 (Pivot Scenarios)

## 개요
피벗은 "실패"가 아니라 "학습에 기반한 방향 수정"이다. 그러나 모든 방향 변경이 피벗은 아니며, 근거 없는 피벗은 팀을 소진시키고 자원을 낭비한다. AI 스타트업에서 피벗은 더 빠르게 발생하는 경향이 있는데, 기술 변화 속도가 빠르고 PMF 탐색 기간이 짧기 때문이다. 중요한 것은 언제 피벗할지, 어떤 방향으로 피벗할지를 사전에 정의하는 것이다.

---

## 1. 피벗 종류표

| 피벗 유형 | 설명 | 유지되는 것 | 바뀌는 것 | AI 스타트업 예시 |
|---|---|---|---|---|
| **고객 세그먼트 피벗** | 같은 제품으로 다른 고객을 타겟 | 제품/기술 | 타겟 시장 | B2C 영어 학습 앱 → B2B 기업 영어 교육 |
| **문제 피벗** | 같은 고객의 다른 문제를 해결 | 타겟 고객 | 해결하는 문제 | 발음 교정 → 영어 면접 준비 |
| **기능 피벗** | 부가 기능이 핵심 기능이 됨 | 고객/시장 | 핵심 가치 제안 | AI 학습 기록 → AI 복습 스케줄러 |
| **기술 피벗** | 다른 기술 스택으로 동일 문제 해결 | 문제/고객 | 기술 접근법 | 룰베이스 추천 → LLM 기반 추천 |
| **채널 피벗** | 동일 제품을 다른 유통 채널로 | 제품/고객 | 영업/배포 방식 | 앱스토어 → API 파트너십 |
| **수익 모델 피벗** | 같은 제품의 수익화 방식 변경 | 제품/고객 | 비즈니스 모델 | 구독 → 사용량 기반 과금 |
| **플랫폼 피벗** | 컴포넌트에서 플랫폼으로 확장 | 핵심 기술 | 제품 범위 | AI 요약 도구 → AI 문서 워크플로 플랫폼 |

## 2. 피벗 신호: 해야 할 때 vs 버텨야 할 때

### 피벗이 필요한 신호 (피벗 고려)

| 신호 | 기준 | 의미 |
|---|---|---|
| D30 리텐션 지속 하락 | 3개 코호트 연속 5% 미만 | 핵심 가치 미전달 |
| 사용자 이탈 원인이 동일 | 이탈 인터뷰 60%+ 같은 이유 | 구조적 문제 |
| CAC > LTV 지속 | 3개월 이상 CAC가 LTV 초과 | 단위경제 불성립 |
| 세일즈 사이클 비정상 장기화 | B2B 계약까지 6개월+ | 가치 제안 불명확 |
| 핵심 기능 사용률 20% 미만 | 가입자 중 핵심 기능 미사용 | 문제 미스매치 |

### 버텨야 하는 신호 (피벗 자제)

| 신호 | 의미 |
|---|---|
| "좋긴 한데 나중에 쓸게요" | 타이밍 문제, PMF 탐색 계속 |
| 경쟁사가 잘 된다 | 방향 맞음, 실행 집중 필요 |
| 팀 피로감/지루함 | 창업자 편향, 데이터 기반 판단 필요 |
| 투자자가 다른 방향을 원함 | 증거 없는 외부 압력 |
| 기술 트렌드 변화 | 기술 변화 ≠ 고객 문제 변화 |

## 3. AI 스타트업 흔한 피벗 패턴

### 패턴 1: B2C → B2B 피벗

```
배경: B2C 사용자 획득 비용이 높고 리텐션이 낮음
신호: CAC $20 이상, D30 리텐션 < 10%
피벗: 동일 AI 기술을 기업 고객에게 API/화이트라벨로 제공

예시:
  일반 소비자용 AI 문서 요약 앱 (MAU 2,000명, ARPU 3,000원)
  → 법무법인/컨설팅사 대상 AI 계약서 분석 SaaS (고객 20개사, ARPU 100만원)

주의: B2B 피벗 시 세일즈 역량 보완 필요, 제품보다 영업이 핵심
```

### 패턴 2: 도구 → 플랫폼 피벗

```
배경: 단일 기능 도구의 성장이 정체되고, 주변 생태계 수요가 발생
신호: 사용자가 다른 도구와 연동 요청을 반복적으로 요구
피벗: 핵심 기능을 유지하며 외부 개발자/파트너가 연동할 수 있는 API/SDK 공개

예시:
  AI 발음 교정 앱 → 영어 학습 앱 개발사를 위한 발음 분석 API 플랫폼

주의: 플랫폼 피벗은 PMF 확인 후 진행, PMF 전 플랫폼화는 방향 산만
```

### 패턴 3: 기능 피벗 (부가 기능이 핵심으로)

```
배경: 원래 핵심 기능보다 부가 기능의 사용률이 3배 이상 높음
신호: 분석 대시보드에서 예상치 못한 기능의 사용 집중 발견
피벗: 그 기능을 중심으로 제품 재설계

예시:
  AI 영어 회화 앱 → 회화보다 '대화 스크립트 저장/복습' 기능 사용률이 5배
  → AI 영어 표현 암기 앱으로 전환

교훈: 사용 데이터를 정기적으로 분석하면 피벗 방향을 데이터로 결정할 수 있다
```

## 4. 피벗 결정 기준 정의

팀이 사전에 "어떤 조건이 충족되면 피벗을 고려한다"를 합의해야 한다.

```
피벗 결정 프레임워크:

Step 1: 현재 가설 재확인
  └─ 원래 핵심 가설이 무엇이었는가?
  └─ 해당 가설이 검증/기각되었는가?

Step 2: 데이터 기반 진단
  └─ 문제인가? 해결책인가? 타겟인가? 채널인가?
  └─ 어느 연결 고리가 끊어진 것인가?

Step 3: 피벗 최소화 원칙
  └─ 가장 적게 바꾸고 가설을 다시 검증할 수 있는가?
  └─ 전체를 바꾸기 전에 한 가지 변수만 바꿀 수 있는가?

Step 4: 새 가설 정의
  └─ 피벗 후 핵심 가설을 새롭게 1문장으로 정의
  └─ 검증 기간과 성공 기준을 재정의

Step 5: 피벗 실행
  └─ 기존 방향을 완전히 종료 후 새 방향에 집중
  └─ "양다리 피벗" 금지 — 두 방향을 동시에 추구하면 둘 다 실패
```

## 흔한 레드플래그

- **피벗 조건이 사전에 정의되어 있지 않다**: "느낌상 안 되는 것 같다"는 피벗 기준이 아니다. 숫자로 미리 정해둬야 한다.
- **전체를 바꾸는 피벗**: 고객도, 문제도, 기술도, 채널도 다 바꾸면 그건 새 창업이지 피벗이 아니다. 하나씩 바꾸며 원인을 찾아야 한다.
- **투자자 조언에 따른 근거 없는 피벗**: 투자자의 "이쪽이 더 클 것 같다"는 의견은 내 고객 데이터보다 신뢰도가 낮다.
- **팀 피로로 인한 피벗**: 3개월 검증 후 지치면 피벗하고 싶어진다. 데이터로 판단해야 한다.
- **경쟁사 피벗 따라 하기**: 경쟁사가 B2B로 갔다고 우리도 갈 이유가 없다. 내 고객 데이터가 기준이다.
- **"AI 트렌드가 바뀌었으니 피벗"**: 기술 트렌드 변화는 기술 업데이트 이유는 되지만, 피벗 이유는 되지 않는다.
