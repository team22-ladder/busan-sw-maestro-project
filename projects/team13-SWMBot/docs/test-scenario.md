# 기획서 검증 에이전트 — 테스트 시나리오

팀원이 처음 받아서 바로 실행할 수 있도록 작성한 가이드입니다.

---

## 환경 준비

### 1. 패키지 설치

```bash
# 백엔드
pip install -r requirements.txt

# 프론트엔드
cd frontend
npm install
```

### 2. `.env` 파일 생성 (프로젝트 루트)

```env
UPSTAGE_API_KEY=your_key
TAVILY_API_KEY=your_key
LANGSMITH_TRACING=true
LANGSMITH_API_KEY=your_key
LANGSMITH_PROJECT=asm-team13-ai-study
LANGSMITH_ENDPOINT=https://api.smith.langchain.com
```

> LANGSMITH_* 는 없어도 동작합니다. 트레이싱만 비활성화됩니다.

### 3. 서버 실행 (터미널 2개)

```bash
# 터미널 1 — 백엔드
uvicorn backend.main:app --reload

# 터미널 2 — 프론트엔드
cd frontend && npm run dev
```

브라우저에서 `http://localhost:5173` 접속.

## 시나리오 A — 정상 실행 (백엔드 + API 키 있음)

### 단계 1. 파일 업로드

1. 왼쪽 파일 드롭존에 기획서 파일 드래그 앤 드롭 또는 클릭 선택
   - 지원 형식: `.txt` `.md` `.pdf` `.docx`
2. **심사 시작** 버튼 클릭

**확인 포인트**
- WorkflowStepper가 `업로드` → `검증` 단계로 전환
- 채팅창에 "심사를 시작합니다" 초기 메시지 표시

---

### 단계 2. 사전 검증 (자동, 사용자 입력 불필요)

백엔드가 자동으로 두 가지 검증을 수행합니다.

#### 2-1. 정적 체크리스트 (`verification`)

오른쪽 **검증 탭**에 7개 항목이 표시됩니다.

| 항목 | 기대 상태 |
|------|-----------|
| 출처 검증 | pass / warn / fail |
| BM 명확성 | pass / warn / fail |
| 문제 구체성 | pass / warn / fail |
| 차별화 근거 | pass / warn / fail |
| 기술 실현성 | pass / warn / fail |
| MVP 범위 | pass / warn / fail |
| 팀 적합성 | pass / warn / fail |

**확인 포인트**
- 7개 항목 전부 표시
- 각 항목에 `reason` 텍스트 있음
- pass(녹색) / warn(노랑) / fail(빨강) 색상 구분

#### 2-2. 수치 데이터 검증 (`data_verification`)

기획서에서 수치 주장(TAM, 성장률, 사용자 수 등)을 최대 4개 추출해 Tavily 웹 검색으로 팩트체크합니다.

**확인 포인트**
- 검증 탭 하단에 수치 검증 결과 카드 표시
- 각 항목: 주장 원문 + `confirmed` / `unconfirmed` / `contradicted` 상태 + 출처
- 수치 주장이 없는 기획서는 이 섹션이 비어 있어도 정상

---

### 단계 3. 심사 라운드 (투자자 → CTO → 멘토)

검증이 끝나면 첫 번째 페르소나가 질문을 스트리밍합니다.

#### 3-1. 질문 수신

- 왼쪽 채팅에 페르소나 뱃지(`💼 깐깐한 투자자` 등)와 함께 질문 말풍선 표시
- 글자가 실시간으로 타이핑되듯 스트리밍
- 질문은 `?` 한 개로 끝남 (이후 내용은 서버에서 잘림)

**확인 포인트**
- 스트리밍 도중 입력창이 비활성화(disabled)
- 스트리밍 완료 후 입력창 활성화

#### 3-2. 답변 입력

- 하단 입력창에 답변 작성 후 **전송** (Enter 또는 버튼 클릭)

**확인 포인트**
- 내 답변이 오른쪽 정렬 말풍선으로 표시
- 전송 직후 `followup_judge` 판정이 오른쪽 **답변 탭**에 표시
  - 점수 (0~100), 임계값, 꼬리질문 여부, 판단 이유

#### 3-3. 꼬리 질문 (조건부)

| 상황 | 임계값 |
|------|--------|
| 첫 번째 답변 | score < 30 → 꼬리 질문 |
| 첫 번째 꼬리 답변 | score < 15 → 두 번째 꼬리 질문 |
| 두 번째 꼬리 답변 | score < 5 → 세 번째 꼬리 질문 |
| 이후 | 항상 다음 라운드 진행 |

**확인 포인트**
- 꼬리 질문은 앞선 질문과 핵심 주제가 달라야 함 (같은 주제 반복 금지)
- 답변이 충분하면 (score ≥ 임계값) 다음 페르소나로 넘어감
- 내 말풍선에 `AnswerFeedbackChip`으로 점수가 표시됨

#### 3-4. 페르소나 순서 반복

- 투자자(💼) → CTO(💻) → 멘토(🦉) 순서로 진행
- 각 페르소나는 최대 2회 꼬리 질문까지 포함해 라운드 진행
- WorkflowStepper가 `심사` 단계 pulse animation으로 유지

---

### 단계 4. 최종 리포트

모든 라운드 종료 후 `reporter` 노드가 구조화된 JSON 리포트를 생성합니다.

**확인 포인트**
- 오른쪽 패널이 **리포트 탭**으로 자동 전환
- WorkflowStepper가 `리포트` 단계로 완료 표시
- 리포트 구성:
  - 종합 완성도 점수 (0~100)
  - 전반적 평가 (2~3문장 요약)
  - 취약점 목록: 섹션명 / 문제점 / 위험도(상·중·하) / 위험 점수 / 개선 제안
  - 마무리 제언
- 입력창은 완전히 비활성화

## 시나리오 B — 데모 모드 (백엔드 없이 UI 확인)

백엔드 서버를 끄거나 API 키 없이 업로드를 시도하면 자동으로 데모 모드로 전환됩니다.

### 실행 방법

1. 백엔드를 끈 상태에서 파일 업로드 시도
2. 또는 `.env` 없이 실행 후 업로드 시도

### 확인 포인트

- 채팅 헤더에 `데모 모드` 배지 표시
- Mock 검증 결과 3개 (문제 구체성·BM 명확성·MVP 범위) 표시
- Mock 수치 검증 2개 표시
- 투자자·CTO·멘토 Mock 질문 순서대로 스트리밍
- 답변 입력 → 다음 Mock 질문 진행
- 마지막 답변 후 Mock 리포트 표시 (overall_score: 67)

> 데모 모드는 API 키 없이 UI 전체 흐름을 확인할 때 사용합니다.

## 시나리오 C — 엣지 케이스

### C-1. 짧은 답변 ("네", "모르겠어요")

- 짧은 답변도 LLM이 질문 맥락을 고려해 점수를 판단
- 맥락상 충분하면 score ≥ 임계값 → 다음 라운드 진행
- 맥락상 불충분하면 꼬리 질문 발생

**확인 포인트**: 단순 가드레일(글자 수 기준 거부)이 없어야 함

### C-2. 수치 없는 기획서 업로드

- `data_verification` 결과가 0건 → 검증 탭 하단 수치 섹션 비어 있음

### C-3. 지원하지 않는 파일 형식

- `.hwp`, `.xlsx` 등 업로드 시도
- 드롭존에서 거부 + 에러 메시지 표시 (`TXT, MD, PDF, DOCX 파일만 지원합니다.`)

### C-4. 중복 질문 방지 확인

- 여러 라운드를 진행하면서 페르소나가 앞선 질문과 핵심 주제·키워드가 겹치는 질문을 내지 않는지 확인
- 표현만 바꾼 동일 질문도 금지되어야 함

## 오른쪽 패널 탭별 확인 항목

| 탭 | 언제 활성화 | 확인 내용 |
|----|------------|-----------|
| **검증** | 업로드 직후 | 정적 체크리스트 7개 + 수치 검증 결과 |
| **답변** | 각 답변 후 | followup_judge 점수·임계값·이유·꼬리질문 여부 |
| **리포트** | 세션 종료 후 | overall_score·취약점·closing |
| **로그** | 전 구간 | 모든 debug 이벤트 원시 기록 |

> 탭은 수신 이벤트에 따라 자동 전환됩니다 (리포트 수신 → 리포트 탭, 답변 판정 → 답변 탭).

## 자주 나오는 질문

**Q. 업로드는 되는데 질문이 안 나와요.**
→ 백엔드 터미널 확인. `uvicorn backend.main:app --reload` 실행 중인지 확인하고, `.env`에 `UPSTAGE_API_KEY`가 있는지 확인.

**Q. 질문이 중간에 잘리거나 두 번 나와요.**
→ 브라우저 개발자 도구 Console 탭 확인. 에러 메시지가 있으면 공유.

**Q. 수치 검증 결과가 안 나와요.**
→ `.env`에 `TAVILY_API_KEY`가 있는지 확인. 없으면 수치 검증 단계를 건너뜁니다.

**Q. 데모 모드로만 진행돼요.**
→ 백엔드가 실행 중인지, API 키가 설정됐는지 확인. 둘 다 정상이면 브라우저 Network 탭에서 `/upload` 요청 실패 여부 확인.
