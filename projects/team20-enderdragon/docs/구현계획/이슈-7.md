## 1. 문제 / 목표

**문제**: 챗봇이 "광물 → 최소 곡괭이 티어", "재료 → 결과물" 관계를 사실이 아닌 **추측으로 생성**해 자기모순·날조가 발생한다.
- `"철은 어떻게 캐?"` → 헤드라인 "철 곡괭이 필요"(❌ 정답=돌 곡괭이) ↔ 본문 "돌 곡괭이로 캐기" → **자기모순** (5/5 재현)
- `"양털 어떻게 구해?"` → 가위 경로를 "돌 → 철 주괴 제작"으로 **날조**

**목표**: 핵심 게임 사실(채굴 티어 + 주요 제작/획득 레시피)을 **권위 있는 ground truth로 강제 주입**해 환각을 제거한다.

---

## 2. 원인 (검증 완료)

위키에 정보는 있으나:
1. **핵심이 '표'에 있어 적재 시 붕괴** — `아이템/곡괭이.md` 제작표가 `[[판자]][[판자]]…`(수십 개) 노이즈로 뭉개지고, "곡괭이의 **[티어]**는…"에서 핵심 단어가 누락됨.
2. **RAG가 오답 유도** — "철" 질문에 글자 매칭으로 **"철 곡괭이" 페이지가 상위**로 옴. 정작 "철광석=돌 곡괭이"는 표에 묻혀 안 떠옴.
3. **사실이 여러 페이지에 흩어짐** → top-5 한 번에 정확한 사실이 안 모임.

→ RAG만으론 불안정. **정적 사실 주입**이 정답.

---

## 3. 접근 (Approach A)

**정적 사실 모듈 + 권위 주입.** 런타임에 안 바뀌는 티어·레시피를 버전관리되는 코드 모듈에 두고, 검색 단계에서 질문에 해당하는 확정 사실을 찾아 responder 프롬프트에 **"위키보다 우선하는 확정 규칙"**으로 주입한다.

> **재사용 원칙**: 그래프 노드 구조(analyze→retrieve→respond) 유지. `search_documents()` 시그니처 무수정. 검색 노드에 사실 주입만 더한다.

## 4. 데이터 모델

`app/knowledge/minecraft_facts.py` (정적·버전관리). **티어는 채굴 '레벨'(숫자) 기반** — 금 곡괭이(레벨 0)가 철·다이아를 못 캐는 걸 정확히 표현한다.

```python
# 곡괭이별 채굴 레벨 (★금 곡괭이는 0 = 나무와 동급)
PICKAXE_LEVEL = {"나무": 0, "금": 0, "돌": 1, "철": 2, "다이아몬드": 3, "네더라이트": 4}

# 광물/블록 → 필요한 최소 채굴 레벨
MINING_REQ = {
    "돌": 0, "석탄 광석": 0,
    "철광석": 1, "구리 광석": 1, "청금석 광석": 1,
    "금광석": 2, "레드스톤 광석": 2, "다이아몬드 광석": 2, "에메랄드 광석": 2,
    "흑요석": 3, "고대 잔해": 3,
}

# 레벨 → 사실 문자열 (금/나무 제외를 명시해 환각 차단)
LEVEL_PHRASE = {
    0: "아무 곡괭이로 채굴 가능",
    1: "돌 곡괭이 이상 필요 (나무·금 곡괭이로는 못 캠)",
    2: "철 곡괭이 이상 필요 (★금 곡괭이 제외)",
    3: "다이아몬드 또는 네더라이트 곡괭이 필요",
}

# 제작 레시피 (작업대 조합)
RECIPES = {
    "나무 곡괭이": "판자 3 + 막대기 2",
    "돌 곡괭이":   "조약돌 3 + 막대기 2",
    "철 곡괭이":   "철 주괴 3 + 막대기 2",
    "다이아몬드 곡괭이": "다이아몬드 3 + 막대기 2",
    "가위":       "철 주괴 2",
    "침대":       "양털 3개(★같은 색) + 판자 3",
    "화로":       "조약돌 8개 (또는 흑암·심층암 조약돌 등 '돌 등급' 블록 8개)",
    "작업대":     "판자 4",
    "막대기":     "판자 2",
    "횃불":       "석탄 또는 목탄 1 + 막대기 1",
}

# 획득/가공 사실 (제작이 아니거나 경로가 헷갈리는 것)
ACQUIRE = {
    "철 주괴": "주로 '철 원석(raw iron)'을 화로/용광로에 제련해 얻음 "
              "(철광석 채굴 시 철 원석이 떨어짐; 섬세한 손길로 캔 철광석 블록도 제련 가능). "
              "철 블록(→9개)·철 조각(9개→1)으로 제작도 가능.",
    "다이아몬드": "다이아몬드 광석을 철 곡괭이 이상(★금 곡괭이 제외)으로 채굴해 획득.",
}

# 검색 매칭용 별칭
ALIASES = {"철": "철광석", "철 광석": "철광석", "다이아": "다이아몬드 광석",
           "다이아몬드": "다이아몬드 광석", "금": "금광석", "구리": "구리 광석"}
```

> 총 ~25-30개 항목. **리뷰 교정 반영분**: ①티어=레벨 기반(금=0) ②침대 같은 색 ③화로=돌 등급 블록 8 ④철 주괴=철 원석 제련+제작도 가능 ⑤"이상"에 금 곡괭이 제외.
> 출처: [Pickaxe](https://minecraft.wiki/w/Pickaxe) · [Golden Pickaxe](https://minecraft.wiki/w/Golden_Pickaxe) · [Iron Ingot](https://minecraft.wiki/w/Iron_Ingot) · [Raw Iron](https://minecraft.wiki/w/Raw_Iron) · [Bed](https://minecraft.wiki/w/Bed) · [Furnace](https://minecraft.wiki/w/Furnace)

> 참고: 이후 이슈 #18에서 이 테이블에 `ITEM_ID_TO_KO`(아이템 ID→한국어명)가 추가되어 인벤토리 인식에도 재사용된다.

## 5. 통합 지점 (데이터 흐름)

```
analyze → retrieve ─┬─ 벡터검색(Qdrant)        → search_results
                    └─ lookup_facts(키워드)     → structured_facts   ← 신규
        → respond (프롬프트에 [확정 규칙]으로 structured_facts 주입, 위키보다 우선)
```

- `lookup_facts(query, keywords) -> list[str]` — 키워드·별칭을 `MINING_REQ`/`RECIPES`/`ACQUIRE`와 매칭해 **완성된 사실 문장** 리스트 반환(티어는 `LEVEL_PHRASE`로 변환).
- responder 프롬프트 맨 앞 `[확정 규칙]` 섹션으로 주입 + "위키와 충돌 시 이 규칙 우선" 지침.

## 7. 기술 메모 / 엣지 케이스

- **금 곡괭이 예외**: 금 곡괭이는 등급상 '나무'와 동급(철 이상 못 캠). 티어 표·`LEVEL_PHRASE`에 명시.
- **주입 우선순위**: responder 프롬프트에서 `[확정 규칙]`을 검색 본문보다 위에 배치하고 "충돌 시 우선" 명시.
- **별칭/표기 흔들림** ("철"/"철광석"/"철 광석") → 부분매칭 + `ALIASES`로 흡수.
- **best-effort**: 매칭 사실이 없으면 빈 리스트 → 기존 위키 기반 답변 그대로(회귀 없음).
- **결정적 테스트**: LLM 최종 문장이 아니라 **주입 단계(`structured_facts` 내용)**를 검증해 플레이키 방지. 라이브 챗은 수동 보조 검증.
