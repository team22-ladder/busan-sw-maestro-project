## 1. 전체 아키텍처

마인크래프트 초보자의 **현재 상황(자원·인벤토리·진척도)**을 파악해 목표까지의 **"다음 한 걸음"**을 안내하는 LLM 에이전트다. 두 종류의 클라이언트가 **하나의 백엔드 API**를 공유한다.

```
[웹 챗봇(Streamlit, 검증용)]  ─┐
                              ├─▶ FastAPI (API · SSE)
[Fabric Mod(인게임, 메인)]  ──┘        │
                                       ├─▶ LangGraph (Agentic · RAG)
                                       │       ├─ MySQL  (세션 · 메시지)
                                       │       ├─ Qdrant (Vector · RAG)
                                       │       └─ Upstage Solar (LLM · 임베딩)
                                       └─▶ knowledge/ (확정 사실 — 환각 방지)
```

> 전체 그림은 [01-시스템-아키텍처.png](img/01-시스템-아키텍처.png), 목표로 하는 11노드 흐름은 [02-에이전트-흐름-전반부.png](img/02-에이전트-흐름-전반부.png) · [03-에이전트-흐름-후반부.png](img/03-에이전트-흐름-후반부.png) 참고.

**설계 원칙**: 백엔드는 클라이언트에 종속되지 않는다. 웹뷰든 Fabric 모드든 동일한 `POST /api/v1/chat*` 계약만 지키면 된다.

### 기술 스택

| 영역 | 기술 | 적용 |
| --- | --- | --- |
| API 서버 | **FastAPI** + Uvicorn | `/api/v1/chat`(SSE), `/chat/sync`(동기), 세션 CRUD |
| 에이전트 | **LangGraph** | 상태 그래프 기반 5노드 워크플로우 |
| LLM | **Upstage Solar** (`solar-pro2`) | 질문 분석·되묻기 판단·코칭 응답 |
| 임베딩 | **Upstage** (`solar-embedding-1-large`) | 위키 적재·질의 임베딩 (차원 동적 측정) |
| 벡터 DB | **Qdrant** (Cloud 공유) | 위키 RAG 컬렉션 `minecraft_knowledge` |
| RDB | **MySQL 8** + SQLAlchemy 2 | users / chat_sessions / messages |
| 마이그레이션 | **Alembic** | 스키마 버전 관리(팀 동기화) |
| 검증 UI | **Streamlit** | 챗봇 동작 검증용 웹뷰 |
| 인게임 | **Fabric** (MC 1.21.1 · Java 21) | 실제 메인 클라이언트 모드 |
| 런타임 | Python 3.12 · `uv` · docker-compose | 의존성·로컬 인프라 |

## 2. LangGraph 워크플로우 (5노드)

`app/graph.py`가 상태 그래프를 정의한다. 상태 타입은 `app/schemas.py`의 `AgentState`(TypedDict).

```
START → analyze ─(domain)─┬─ general·out_of_scope ─────────────→ respond → END
                          └─ minecraft → clarify ─(정보 충분?)─┬─ 부족 → ask → END (되묻기)
                                                               └─ 충분 → retrieve → respond → END
```

| 노드 | 파일 | 역할 |
| --- | --- | --- |
| `analyze` | `agents/query_analyzer.py` | domain(`minecraft`/`general`/`out_of_scope`)·keywords·intent를 structured output으로 추출. 파싱 실패 시 `minecraft` 기본값으로 폴백 |
| `clarify` | `agents/clarifier.py` | 정보가 부족하면 되묻기 질문 생성 |
| `ask` | `graph.py` | 되묻기 질문을 최종 답변으로 반환하고 종료 |
| `retrieve` | `agents/retrieval.py` | Qdrant 위키 검색 + `knowledge/`의 확정 사실 병행 조회 |
| `respond` | `agents/responder.py` | 도메인별 응답 생성 (minecraft는 토큰 스트리밍) |

**분기 규칙**
- `route_by_domain`: `minecraft`만 `clarify`로, 그 외(`general`/`out_of_scope`)는 곧바로 `respond`.
- `route_by_clarification`: 되묻기 필요 시 `ask`(종료), 아니면 `retrieve`.

**도메인별 응답 전략** (`responder.py`)
- `out_of_scope` → 고정 안내 문구 (LLM 호출 없음)
- `general` → temperature 0.5 가벼운 답변
- `minecraft` → temperature 0.2 + `streaming=True`, 위키 검색 결과·확정 사실·인벤토리를 모두 프롬프트에 주입

### 되묻기(clarification)와 무한 루프 방지 — 이슈 #11

`clarify` 노드는 "지금 뭐 해야 해?"처럼 상황 정보가 없으면 답이 불가능한 질문에 되묻는다. 단, 무한 되묻기를 막기 위해 **상태 기반**으로 두 경우를 스킵한다.

- **인벤토리가 있으면** 보유 아이템으로 상황이 파악되므로 되묻지 않는다.
- **직전 턴이 되묻기였으면** 스킵한다. 되묻기 문구가 `?`로 끝나지 않는 경우가 많아 텍스트가 아닌 **메시지 role(`clarification`)**로 판정한다(`api.py`의 `prev_was_clarification`).

> 내부적으로 되묻기 메시지는 `messages` 테이블에 `role='clarification'`으로 저장하되, 프론트엔드에는 `assistant`로 노출한다.

## 3. RAG — 위키 코퍼스와 Qdrant (이슈 #5)

### 코퍼스
- 출처: ko.minecraft.wiki를 Obsidian vault로 정리한 마크다운(파일별 YAML 프론트매터 + 본문).
- 규모: 본문 약 1,225개 → 청킹 후 **약 5,923개 벡터**.

### 적재 파이프라인 — `scripts/ingest_wiki.py`
1. vault를 순회하며 인덱스/메타 디렉토리(`00-인덱스`, `_meta`, `.obsidian` 등) 제외.
2. 본문 정리: `[[위키링크|표기]]→표기`, 반복 토큰(제작표 노이즈) 압축, 초대형 페이지 상한(20,000자).
3. `RecursiveCharacterTextSplitter`로 청킹.
4. Upstage 임베딩 → Qdrant upsert. payload에 `content`, `title`, `category`, `source_url` 등 메타 저장.

```bash
uv run python scripts/ingest_wiki.py --limit 50   # 시범(속도 보정)
uv run python scripts/ingest_wiki.py              # 전체 적재
```

### 검색 — `app/vector_store.py`
- `search_documents(query, n_results)` → `[{content, metadata, distance}]` (distance에 Qdrant 유사도 score, 높을수록 유사).
- 컬렉션은 Cosine 거리, 차원은 `embedding_dim()`이 샘플 임베딩으로 동적 측정(모델 교체에 견고).

### 협업 모델
- **위키 벡터는 공유 Qdrant Cloud에 1회 적재** → 팀원은 `QDRANT_URL`/`QDRANT_API_KEY`로 연결만 하면 재임베딩이 필요 없다.
- MySQL은 각자 **로컬**(세션은 일회성 개발 데이터).

## 4. 환각 방지 — 확정 사실 계층 (이슈 #7)

RAG만으로는 채굴 곡괭이 티어·제작 재료 같은 **정확한 수치**에서 LLM이 환각을 일으킨다(예: "돌→철 주괴" 날조). 이를 막기 위해 위키 검색과 별개로 **하드코딩된 확정 사실 계층**을 둔다.

- `app/knowledge/minecraft_facts.py` — 테이블 정의
  - `MINING_REQ` / `PICKAXE_LEVEL` / `LEVEL_PHRASE` — 광물별 필요 곡괭이 티어
  - `RECIPES` — 제작 레시피
  - `ACQUIRE` — 획득/가공법 (중간재 포함)
  - `ALIASES` — 별칭 → 표준 키
  - `ITEM_ID_TO_KO` — `minecraft:item_id` → 한국어명
- `app/knowledge/facts.py` — `lookup_facts(query, keywords, inventory_items)`가 질문·키워드·인벤토리에서 위 테이블을 조회해 확정 사실 문장 리스트를 반환.

**적용 방식** (`responder.py` + `prompts/templates.py`)
- 확정 사실을 `[확정 규칙]` 블록으로 만들어 **참고 위키보다 위에** 주입하고, 충돌 시 확정 규칙을 우선하도록 시스템 프롬프트에 명시한다.
- 보유 재료가 레시피의 재료로 쓰이면 그 중간재의 획득법까지 함께 주입해 날조를 차단한다.

## 5. 데이터 계층 — MySQL (이슈 #5)

### 스키마 — `app/models.py`
| 테이블 | 주요 컬럼 | 설명 |
| --- | --- | --- |
| `users` | id, external_id, created_at | 초기엔 익명/기본 유저 |
| `chat_sessions` | id, thread_id(unique), user_id(FK), created_at | `thread_id`가 대화 맥락 단위 |
| `messages` | id, session_id(FK), role, content, created_at | role: `user`/`assistant`/`clarification`(내부용) |

- 스키마는 **Alembic 마이그레이션**으로 버전 관리 → 팀원은 `uv run alembic upgrade head`로 동기화.
- 모델 변경 시 `uv run alembic revision --autogenerate -m "..."` → 커밋.

### 데이터 접근 계층 — `app/repositories.py`
DB 내부를 몰라도 아래 함수만 호출하면 되는 **팀 공용 인터페이스**.
- `get_or_create_session(db, thread_id)` · `append_message(db, session_id, role, content)` · `get_recent_messages(...)` · `list_sessions(...)` · `delete_session(...)`

### best-effort 저장 — 이슈 #11
- 대화 이력 로드/저장(`_load_history`, `_save_turn`)은 예외를 삼킨다. **DB 장애가 응답을 막지 않는다.**

## 6. API 계약

`app/main.py`가 라우터를 `/api/v1` prefix로 마운트한다.

| 엔드포인트 | 설명 |
| --- | --- |
| `POST /api/v1/chat/sync` | 동기 → `{ answer, domain, sources }` — **Fabric 모드**가 사용 |
| `POST /api/v1/chat` | SSE 스트리밍 (노드 진행 이벤트 + 토큰 + `done`) — **웹뷰**가 사용 |
| `GET /api/v1/sessions` | 최근 대화 목록(미리보기 포함) |
| `GET /api/v1/sessions/{thread_id}/messages` | 세션 전체 메시지 |
| `DELETE /api/v1/sessions/{thread_id}` | 세션·메시지 삭제(cascade) |
| `GET /health` | 상태 확인 |

**요청 본문**: `{ message, thread_id, inventory[] }`
- `thread_id`로 세션을 식별·저장하고, 서버는 최근 `HISTORY_TURNS(6)`턴을 프롬프트 맥락으로 사용한다.
- `inventory`는 Fabric 모드만 채운다(웹뷰는 항상 `[]`).

### SSE 스트리밍 이벤트 (이슈 #9 · #11)
`POST /api/v1/chat`는 `graph.astream_events`를 구독해 다음 이벤트를 흘려보낸다.
- `event: "node"` — 각 노드 완료 시 진행 상황(analyze/clarify/retrieve/respond/ask)
- `event: "token"` — `respond` 노드의 LLM 토큰(점진 출력)
- `event: "done"` — 최종 `{ answer, domain, sources }`

> 노드(LLM/벡터DB) 예외로 스트림이 끊겨도 `done`과 DB 저장은 보장한다(이슈 #11에서 SSE 예외 처리 보강).

## 7. 인게임 인벤토리 파이프라인 (이슈 #18)

Fabric 모드가 호출 시점의 플레이어 인벤토리를 캡처해 백엔드 답변 정확도를 높인다.

1. **캡처** (`InventorySnapshot.capture`) — main(36)·armor(4)·offhand(1) 슬롯을 순회해 빈 슬롯을 제외하고 `{ item: "minecraft:cobblestone", count: 3 }` 형태로 수집.
2. **전송** — `CoachApiClient`가 `chat/sync` 요청 본문 `inventory[]`에 실어 보냄.
3. **활용** (백엔드)
   - `retrieval.py`: 아이템 ID를 한국어명으로 바꿔 검색 질의를 보강("…제작법 레시피").
   - `facts.py`: 보유 재료가 쓰이는 레시피·중간재 획득법까지 확정 사실로 조회.
   - `prompts/templates.py`의 `format_inventory_block`: `[현재 인벤토리]` 블록으로 프롬프트에 주입 → "보유 재료로 당장 만들 수 있는 것 우선" 안내.

## 8. 인게임 할 일(TODO) HUD (이슈 #20)

코치 답변의 단계별 안내를 인게임 UI로 끌어온다.

- **파싱** (`TodoList.parseAndAdd`): 답변에서 `- ` 불릿 줄을 추출하고 "N단계:" 접두사·`[위키]`·`(참고…)` 노이즈를 제거. 각 항목을 `shortText`(HUD 요약)·`fullText`(목록 원문) 이중 구조로 저장.
- **HUD** (`TodoHudRenderer`): 화면 우측 상단에 `shortText` 목록을 항상 렌더링(인벤토리 화면 등에선 숨김).
- **목록 화면** (`TodoScreen`, J키): 전체 항목을 보고 ↑/↓ 선택 + Enter로 완료. 완료 시 sparkle 애니메이션·사운드.
