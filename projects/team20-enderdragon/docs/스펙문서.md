## 1. 개요

- **한 줄 정의**: 마인크래프트를 막 시작한 초보자의 현재 상황(시간대·자원·진척도)을 파악해, 사용자가 정한 목표까지 가는 **"다음 한 걸음"** 을 우선순위에 따라 단계별로 안내하는 LLM 기반 Agent.
- **서비스 정의**: "마인크래프트를 처음 켰는데 막막할 때, 검색·정리 없이 내 상황에 딱 맞는 '다음 한 걸음'을 바로 알려주는 게임 길잡이."
- **핵심 문제**: 정보가 부족한 게 아니라 **너무 많고 흩어져 있어서**, 초보자가 "지금 내 상황에서 뭐부터 해야 하는지" 판단하지 못해 초기 이탈한다. (정보 과잉 · 높은 탐색 비용 · 맥락 부재)
- **핵심 가치**: 지식의 양이 아니라 **"다음 한 걸음의 명확함"**. 백과사전이 아니라 길잡이.

### 대상 사용자 (페르소나)
| 유형 | 설명 | 대표 질문 |
| --- | --- | --- |
| 수동형 (길 잃은 입문자) | 14세, 친구 따라 설치. 무엇부터 할지 전혀 모름 | "지금 뭐 해야 해?" (막연형 → Agent가 목표 제안) |
| 능동형 (목표는 있지만 막힌 입문자) | 19세, 영상 보고 작은 목표 생김. 순서를 모름 | "철 곡괭이 만들고 싶은데 돌 곡괭이밖에 없어" (목표형) |

> 두 경우 모두 **"현재 상태 → 목표까지의 단계별 경로"** 라는 같은 엔진으로 처리한다.

## 2. 목표 & 비목표

### 제품 전체 목표 (North Star)
이미지·텍스트로 상황을 인식하고, 목표를 단계별 로드맵으로 분해해, 초보 코치 톤으로 안내하며, 세션 맥락을 유지해 후속 질문에 이어 답하는 11노드 Agentic 시스템.

### 이번 이슈 #5 목표 — "DB 세팅 + 웹뷰 기반 개발"
에이전트 두뇌가 아니라 **데이터 기반(DB + 위키 코퍼스) + 개발/검증 환경의 토대**를 완성한다. 기존 의료 템플릿을 마인크래프트로 갈아끼우고, 새 데이터 스택(MySQL + Qdrant) 위에서 챗봇이 실제 위키 데이터로 end-to-end 동작함을 검증한다. **다른 팀원이 이 토대 위에서 기능을 추가/수정할 수 있도록 협업 환경**까지 갖춘다.

### 비목표 (#5에서 하지 않음 — 후속 이슈)
11노드 Agentic Workflow · Vision(스크린샷 분석) · React/Electron 프론트 · Fabric 플러그인 · 세션 컨텍스트 용량 체크/장기 메모리.

## 3. 대표 시나리오

**"철 곡괭이를 만들고 싶은데 어떻게 해?"**
- 상황: 나무 곡괭이 + 통나무 5개 보유, 조약돌 없음
- Agent 응답(로드맵): 목표(철 확보) / 선행조건(돌 곡괭이↑) / 부족자원(조약돌 3) / 하위목표(조약돌→돌곡괭이→철탐색) / 실행계획("먼저 근처 돌에서 조약돌 3개")

> 막연형("이제 뭐 해야 해?")·위급형("밤 오는데 집이 없어")도 동일 흐름.

## 4. 시스템 아키텍처

### 4.1 전체 그림 (North Star)
```
[웹 챗봇 / Fabric 플러그인]  →  FastAPI (API · SSE/WS)  →  LangGraph (Agentic · RAG)
                                                              ├─ MySQL   (세션 · 유저)
                                                              ├─ Qdrant  (Vector · RAG)
                                                              ├─ LLM API (추론)
                                                              └─ Vision API (스크린샷)
```

### 4.2 Agentic Workflow (11노드 · 후속 목표)
```
goal_analyzer → state_collector → state_sufficiency_checker
   └(부족)→ clarification_question_generator → 종료
   └(충분)→ state_interpreter ─┬→ structured_db_retriever (MySQL)
                               └→ vector_db_retriever (Qdrant/RAG)
        → requirement_analyzer → resource_gap_analyzer → subgoal_planner
        → action_planner → plan_verifier
              └(불가능)→ alternative_planner ┐
              └(가능)──────────────────────┴→ coach_response_generator → memory_update
```

### 4.3 #5 완료 후 실제 구성 (이번 범위)
```
[Streamlit 웹뷰(검증용)]  ─┐
                           ├─▶ FastAPI (/api/v1/chat, /chat/sync, SSE)
[Fabric 플러그인] ─────────┘        │
                                    ├─▶ LangGraph (analyze → retrieve → respond, 3노드 — 이후 #7·#11에서 5노드로 확장)
                                    │        └─ retrieve: Qdrant(minecraft_knowledge) + Upstage 임베딩
                                    └─▶ MySQL (users / chat_sessions / messages)
LLM: Upstage Solar (solar-pro2)
```
> **웹뷰는 챗봇 동작 검증용 테스트 UI**이며, 실제 메인 클라이언트는 후속 **Fabric 플러그인**이다. 둘 다 동일한 FastAPI API를 호출하므로 백엔드는 클라이언트 비종속적으로 유지한다.

## 6. 데이터 설계 (#5)

### 6.1 MySQL 스키마 (SQLAlchemy + Alembic)
- `users` — id, (식별자), created_at *(초기엔 익명/기본 유저)*
- `chat_sessions` — id, thread_id, user_id(FK), created_at
- `messages` — id, session_id(FK), role(user/assistant), content, created_at
- 스키마는 **Alembic 마이그레이션**으로 버전 관리 → 팀원은 `alembic upgrade head`로 동기화

### 6.2 Qdrant 벡터 DB
- 컬렉션 `minecraft_knowledge` · 거리 Cosine · 차원은 Upstage `solar-embedding-1-large` 실제 차원을 샘플로 측정
- 런타임은 **공유 Qdrant Cloud**(url+api_key)에 연결, 로컬 docker-compose Qdrant는 오프라인/적재용
- payload: title, category, mc_edition, source_url, chunk_index (프론트매터에서 추출)

### 6.3 RAG 코퍼스 — 마인크래프트 위키 (실측)
- 출처: 사용자가 ko.minecraft.wiki에서 수집해 Obsidian에 정리한 코퍼스 (`~/obsidian/minecraftwiki`, 별도 git repo)
- 규모: **본문 md 1,225개 · 약 467만 자(≈200만 토큰)** → 청킹 시 **약 6,000청크**
- 형식: 파일별 YAML 프론트매터(title·categories·mc_edition·source_url·revision) + 본문(`[[위키링크]]`·표 포함)
- 초보 콘텐츠 풍부: 튜토리얼 폴더("초보자를 위한 가이드", "첫 10분", "둘째 날", "배고픔 관리" 등)
- **적재 범위(#5): 전체** — 1회 적재해 공유 Qdrant Cloud에 업로드(팀원 재임베딩 불필요). 예상 소요 ~3–8분(최대 ~30분), 비용 <$1. 50개 시범 적재로 속도 보정 후 풀런.
- 적재 전처리: `00-인덱스/`(MOC) 제외, 초대형 페이지 분할 상한, `[[링크|표기]]`→표기 정리, 반복 제작표 노이즈 축소, 헤딩 기준 청킹.

## 7. 협업 & 팀원 온보딩 (이 DB를 팀이 어떻게 쓰나)

**원칙**: DB는 파일로 넘기는 게 아니라 **"코드로 정의된 재현 가능한 인프라"** 로 공유된다. 팀원은 커밋된 코드를 받아 실행하면 동일 DB를 재현한다.

**5개 기둥**
1. **인프라 as code** — `docker-compose.yml` → `docker compose up -d`로 동일한 MySQL+Qdrant
2. **스키마 as code** — SQLAlchemy 모델 + **Alembic** → `alembic upgrade head`로 안전 동기화(데이터 보존)
3. **연결 설정** — `.env.example` → `.env`만 채우면 됨(하드코딩 X)
4. **데이터** — 공유 Qdrant Cloud(벡터 1회 적재 후 연결만) + 적재 스크립트(재현용)
5. **데이터 접근 계층** — `repositories.py`/`vector_store.py`의 함수(`get_or_create_session`, `append_message`, `search_documents`)만 호출 → DB 내부를 몰라도 기능 개발 가능

**팀원 첫날 흐름**
```bash
git pull && cp .env.example .env     # 시크릿 입력(MySQL 로컬, Qdrant=공유 클라우드)
docker compose up -d                 # 로컬 MySQL(+선택 Qdrant)
alembic upgrade head                 # 최신 스키마
# 이제 repositories.py / vector_store.py 위에서 기능 개발
```

**로컬 vs 공유**: MySQL은 각자 **로컬**(세션은 일회성, 격리 우선) / Qdrant 위키 벡터는 **공유 클라우드**(재임베딩 방지).

## 8. API 계약 (웹뷰 + 플러그인 공유)

- `POST /api/v1/chat/sync` → `{ answer, domain }` (동기)
- `POST /api/v1/chat` → SSE 스트리밍 (노드별 이벤트 + `done`)
- `GET /health` → 상태
- 요청: `{ message, thread_id }` — `thread_id`로 세션 식별·저장
