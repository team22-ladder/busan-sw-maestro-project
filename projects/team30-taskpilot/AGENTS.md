## 언어 및 커뮤니케이션 규칙

- **기본 응답 언어**: 한국어
- **코드 주석**: 한국어로 작성
- **커밋 메시지**: 한국어로 작성
- **PR 내용**: 한국어로 작성
- **문서화**: 한국어로 작성
- **변수명/함수명**: 영어 (코드 표준 준수)

LLM의 일반적인 코딩 실수를 줄이기 위한 행동 지침이다. 프로젝트별 지침이 있을 경우 본 가이드라인과 병합하여 사용한다.

트레이드오프: 본 지침은 속도보다 신중함에 우선순위를 둔다. 사소한 작업은 상황에 맞게 판단한다.
모든 답변 및 코드 주석 한국어로 대답.

## 프로젝트 Context 확인

- 항상 CONTEXT-MAP.md 파일을 먼저 참고하여 배경 지식을 참고할 것
- 문서 읽는 순서 : CONTEXT-MAP.md -> 사용자가 요구한 기능과 관련된 폴더의 CONTEXT.md-> 사용자가 요구한 기능과 관련된 폴더의 ADR.md -> 코드
- 각 기능 아래에 ADR.md 를 만들어 ADR : 아키텍쳐 결정 이유, 기술 스택을 알기 쉽게 작성할 것
- 백엔드, 프론트는 테스트를 위한 장치일뿐, 핵심은 python 실행파일로 코드가 동작하는 것.
- reference code는 참고용일뿐, import 해서 코드를 그대로 사용하거나 reference 코드를 수정하지 말것. reference code 폴더 아래에 있는 기능이 필요하면 각 기능 폴더 안에 코드를 재작성 할것.

## 작업 시작 체크리스트

1. `CONTEXT-MAP.md` 확인
2. 작업 대상 기능/패키지의 `CONTEXT.md` 확인
3. 작업 대상 기능/패키지의 `ADR.md` 확인
4. 관련 코드 확인
5. 기존 테스트 구조 확인
6. 구현
7. 기능별 테스트 작성 또는 수정
8. 테스트 실행 결과 보고

## git 작업

- 커밋 메시지는 한국어 Conventional Commits 형식인 `type: 한국어 설명`을 사용한다.
- `type`은 변경 목적에 맞는 영문 소문자 키워드(`feat`, `fix`, `docs`, `test`, `refactor`, `chore` 등)를 사용한다.
- PR 내용은 한국어로 작성한다.
- PR 본문에는 `## 요약`, `## 변경 사항`, `## 테스트 결과` 섹션을 포함한다.
- PR 맨 하단에는 코드 변경 이해도 및 플로우를 확인하는 퀴즈 2개를 필수로 작성한다.
